# 写作规范

本规范确保 AgentDevGuide 全系列文章风格统一、结构一致、逻辑连贯。无论从哪一章开始写，都应严格遵守。

---

## 1. 定位与读者

- **读者画像**：有 1-3 年开发经验的应用程序员，知道 Python/JS 基础，但没接触过 LLM 和 Agent
- **写作视角**：应用开发者视角，不是研究员视角。讲"怎么用"和"为什么这样用"，不推数学公式
- **核心原则**：每篇文章回答一个核心问题，读者读完能做出一个东西或建立一个判断

## 2. 文章结构模板

每篇文章必须包含以下结构（顺序固定）：

```markdown
# 文章标题

## 目录

- [概述](#概述)
- [一级标题 1](#一级标题-1)
  - [二级标题 1.1](#二级标题-11)
- [一级标题 2](#一级标题-2)
- [参考链接](#参考链接)

## 概述

<!-- 2-4 句话，回答：这篇文章解决什么问题？为什么重要？读完能得到什么？ -->

## 一级标题 1

<!-- 正文 -->

## 参考链接

<!-- 3-8 条高质量参考，优先官方文档和原始论文 -->
```

### 结构规则

| 规则 | 说明 |
|------|------|
| 必须有目录 | 用 Markdown 锚点链接，方便跳转 |
| 必须有概述 | 开门见山，2-4 句话说明本文解决什么问题 |
| 必须有参考链接 | 放在文末，3-8 条，优先官方文档和原始论文 |
| 概述后直接进正文 | 不要加"让我们开始"之类的过渡 |
| 文末引导下一篇 | 用引用块 `>` 引导读者进入下一篇，形成阅读链 |
| 章节间要有逻辑递进 | 从定义/概念 → 原理/机制 → 实践/应用 → 局限/权衡 |

## 3. 章节目录（README.md）模板

每个章节目录必须包含：

```markdown
# XX — 章节名称

<!-- 1-2 句话：本章在整个体系中的位置，解决什么问题 -->

## 目录

| # | 文章 | 内容 |
|---|------|------|
| 01 | [文章标题](./01-xxx.md) | 一句话概括 |
| 02 | [文章标题](./02-xxx.md) | 一句话概括 |

> 学完本章后，请继续阅读 [XX — 下一章](../xx-next/README.md)，进入 xxx。
```

## 4. 写作风格

### 4.1 语气与人称

| 规则 | 正确 | 错误 |
|------|------|------|
| 用第二人称"你" | "你需要知道" | "读者需要知道"、"我们需要了解" |
| 不用"我们"代指作者+读者 | "接下来看" | "接下来让我们一起来看看" |
| 陈述句为主 | "Transformer 改变了一切" | "Transformer 可以说是改变了一切吧" |
| 避免口语化 | "自回归生成的智能涌现机制" | "为什么预测下一个词就能变聪明" |
| 避免学术腔 | "模型参数越多，能力越强" | "模型性能与参数量呈正相关" |

### 4.2 段落与节奏

- **每段 2-5 句**，超过 5 句必须拆段
- **先结论后解释**：先说"是什么/为什么"，再说"怎么做"
- **关键论点加粗**：用 `**加粗**` 标记每段的核心论点，方便扫读
- **类比用一次就够**：类比帮助理解，但不要反复用同一个类比
- **数字要具体**：说"1750 亿参数"不说"参数量很大"；说"API 价格下降 10 倍"不说"价格大幅下降"

### 4.3 技术概念引入

引入新概念时遵循 **定义 → 举例 → 意义** 三步：

```markdown
**涌现能力（Emergent Abilities）** 是指当模型参数量跨过某个阈值后，
突然具备小模型完全没有的能力。<!-- 定义 -->

比如 GPT-3 在 1750 亿参数时突然学会了做数学推理和写代码，
而 13 亿参数的版本完全做不到。<!-- 举例 -->

这意味着"不够大就不行"——量变引发了质变。<!-- 意义 -->
```

### 4.4 代码示例

- 代码必须有注释，注释用中文
- 代码必须可运行（或明确标注 `# 省略部分代码`）
- 优先用 Python，必要时补充 TypeScript
- 代码块前用一句话说明"这段代码做什么"
- 不要放超过 40 行的代码块，超长代码拆成多段

### 4.5 图片与图表

- 图片居中显示，使用统一格式：

```markdown
<p align="center">
  <img src="../../assets/xx-chapter/filename.png" alt="描述文字" width="90%"/>
  <br/>
  <em>图片说明</em>
</p>
```

- 图片放在 `assets/xx-chapter/` 目录下
- 图片说明用 `<em>` 斜体，简短描述图片内容
- 优先用 PNG 格式，架构图可用 SVG

## 5. 术语规范

| 统一用法 | 不要用 | 说明 |
|---------|--------|------|
| LLM | 大模型、大语言模型、语言大模型 | 首次出现写全称"大语言模型（LLM）"，之后用 LLM |
| Token | 词、字、token | 首次出现写"Token（词元）"，之后用 Token |
| Prompt | 提示词、提示语 | 首次出现写"Prompt（提示词）"，之后用 Prompt |
| Agent | 智能体、代理 | 首次出现写"Agent（智能体）"，之后用 Agent |
| 上下文窗口 | 上下文长度、context window | 首次出现写"上下文窗口（Context Window）" |
| 幻觉 | Hallucination | 首次出现写"幻觉（Hallucination）" |
| 微调 | Fine-tuning | 首次出现写"微调（Fine-tuning）" |
| 工具调用 | Function Calling、函数调用 | 首次出现写"工具调用（Function Calling）" |
| 流式输出 | Streaming、流式传输 | 首次出现写"流式输出（Streaming）" |
| 推理模型 | Reasoning Model | 首次出现写"推理模型（Reasoning Model）" |

### 术语引入规则

- **首次出现**：中文（英文），如"大语言模型（Large Language Model, LLM）"
- **后续出现**：用统一缩写，如 LLM、Token、Agent
- **同一概念在整本书中只引入一次**：第 01 章引入过的术语，后续章节直接使用缩写

## 6. 章节间衔接

### 前向引用（预告）

在当前章节末尾，预告后续章节如何解决当前遗留的问题：

```markdown
> 幻觉用 RAG 解决，计算弱点用工具调用补齐，上下文限制用记忆管理应对。
> 接下来请阅读 [从 NLP 到 Transformer](./02-nlp-to-transformer.md)。
```

### 后向引用（回顾）

在章节概述中，简短回顾前置知识：

```markdown
在 [01 — LLM 基础](../01-llm-basics/README.md) 建立认知之后，
这一章解决动手问题：怎么把模型调通。
```

### 衔接规则

- 每篇文章末尾必须有"下一篇"引导
- 每个章节 README 的概述必须有"上一章"回顾
- 不要重复讲前置章节已经讲过的概念，用链接引用代替

## 7. 内容质量检查清单

写完一篇文章后，逐项检查：

- [ ] 概述是否在 4 句话内说清"解决什么问题"
- [ ] 目录锚点是否与实际标题一致
- [ ] 每个新概念是否按"定义→举例→意义"引入
- [ ] 术语是否遵循统一规范（首次中文+英文，后续缩写）
- [ ] 代码示例是否可运行、有注释、不超过 40 行
- [ ] 图片是否在 `assets/` 目录、格式统一
- [ ] 参考链接是否 3-8 条、优先官方文档
- [ ] 文末是否有下一篇引导
- [ ] 段落是否超过 5 句（超过则拆分）
- [ ] 关键论点是否加粗标记
- [ ] 是否有口语化标题（有则改为技术表述）

## 8. 公众号发布流程

文章写完后，使用 `my-tools/wechat-mp-manage` 工具一键发布到微信公众号。

### 前置条件

- `my-tools` 仓库已 clone 到本地，`wechat-mp-manage` 包已安装（`pip install -e .`）
- 公众号凭据已配置：`wechat-mp-manage config <app_id> <app_secret>`
- Playwright 已安装 Chromium：`python -m playwright install chromium`

### 发布命令

```bash
wechat-mp-manage draft docs/xx-chapter/xx-article.md --series "AI Agent 开发指南" --sub "章节名" --theme <主题>
```

参数说明：

| 参数 | 说明 | 示例 |
|------|------|------|
| `--series` | 系列名（封面左上角） | "AI Agent 开发指南" |
| `--sub` | 子分类（封面居中） | "LLM 基础"、"Agent 循环" |
| `--theme` | 封面配色 | `aurora` / `ocean` / `midnight` / `sunset` / `forest` |

### 发布流程说明

工具内部自动完成以下步骤：

1. **Markdown → HTML**：转换为公众号兼容的内联样式 HTML
   - 自动移除目录、锚点链接、`target="_blank"`
2. **上传正文图片**：检测本地图片路径
   - SVG 自动转换为 PNG（通过 Playwright 截图）
   - 所有图片上传到微信素材库，替换为微信 CDN URL
3. **生成封面图**：根据标题和配色自动生成带系列标识的封面
4. **上传封面**：封面图作为永久素材上传
5. **创建草稿**：调用微信 API 创建图文草稿，返回 `media_id`

### 草稿管理

```bash
wechat-mp-manage list                  # 列出草稿
wechat-mp-manage submit <media_id>     # 发布草稿
wechat-mp-manage delete <media_id>     # 删除草稿
wechat-mp-manage status <publish_id>   # 查询发布状态
```
