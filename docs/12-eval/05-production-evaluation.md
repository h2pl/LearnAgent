你好，我是江小湖。前四篇文章覆盖了"评测什么"（体系与指标）、"确定性方法"（代码检查/执行验证/参考比对）、"灵活方法"（LLM-as-Judge）和"怎么融入开发流程"（EDD）。但 Agent 上线之后，评测并没有结束——**你可能需要问一些离线环境回答不了的问题**：

- 用户真的觉得 Agent 好用吗？
- Agent 的表现随时间在变差吗？
- 离线评测通过的场景，生产环境真的表现一致吗？

这就是生产环境评测要解决的问题。

## 离线评测 vs 在线评测

| 维度 | 离线评测 | 在线评测 |
|------|---------|---------|
| 数据来源 | 人工标注的评测集 | 生产环境真实流量 |
| 覆盖范围 | 有限的用例 | 全量真实场景 |
| 反馈速度 | 即时（运行即出结果） | 有延迟（需要收集数据） |
| 成本 | 标注成本 + 计算成本 | 监控基础设施成本 |
| 信号质量 | 高（人工标注保证质量） | 低（用户行为是间接信号） |
| 主要用途 | 开发阶段质量把关 | 上线后持续监控 |

**两者是互补关系**，不是替代关系。离线评测负责"上线前"，在线评测负责"上线后"。

### 你需要在线评测的信号

不依赖用户主动反馈，从 Agent 行为中自动产生的信号：

```
Agent 执行信号:
  - Token 消耗异常增长 → 可能上下文管理出了问题
  - 工具调用次数剧增 → 可能 Agent 陷入了死循环
  - 多次重试同一工具 → 可能工具返回了异常
  - Agent 主动说"我不知道" → 可能检索或知识不足

用户交互信号:
  - 用户重复同一个问题 → Agent 第一次没解决
  - 用户中断对话 → Agent 的回答不满足需求
  - 用户要求转人工 → Agent 处理不了
  - 用户长时间不回复 → Agent 的回答可能不合理
```

## 用户反馈采集

用户反馈是最直接的在线评测信号。

### 显式反馈

用户主动给出的评价：

```
Agent 回答末尾:
  "这个回答对你有帮助吗？👍 👎"

用户点击 👍 → 正面信号
用户点击 👎 → 负面信号 → 触发日志记录和分析
```

显式反馈的好处是信号明确，坏处是参与率低（通常只有 1-5% 的用户会点）。

### 隐式反馈

从用户行为推断的反馈——参与率高，但信号有噪音：

| 用户行为 | 推测的信号 | 置信度 |
|---------|-----------|--------|
| 用户复制了回答 | 有用 | 高 |
| 用户短时间内重复提问 | 没用 | 高 |
| 用户中断对话后重新开始 | 没用 | 中 |
| 用户快速连续点赞 | 可能是误触 | 低 |
| 用户没有做任何操作 | 中立 | 低（默认行为） |

### 信号加权

不同的反馈信号应该有不同的权重：

```
单次会话的质量评分 =
  显式好评 +1.0
  显式差评 -1.0
  复制回答 +0.5
  重复提问 -0.8
  转人工 -0.9
  中断后重开 -0.5
  ──────────────────
  总分（归一化到 0-1 区间）
```

这个评分可以关联到具体的 Trace，用于筛选"好样本"和"坏样本"，定期补充到离线评测集中。

## 回放测试

回放测试 (Production Replay) 是一种介于离线评测和在线评测之间的技术——**用生产环境的真实数据，跑离线评测的流程**。

### 工作流程

```
生产日志 → 提取 (用户输入, Agent 行为) 对
  → 标注 → 加入评测集
  → 重新运行 Agent → 对比结果
```

具体步骤：

```
1. 从生产日志中收集真实用户输入
2. 对于每个输入，记录：
   - 当时 Agent 的行为（调了哪些工具、输出了什么）
   - 最终结果（用户是否满意、任务是否完成）
3. 人工或自动标注"正确行为"
4. 用新版本的 Agent 重新处理这些输入
5. 对比新旧版本的差异
```

### 回放的价值

**发现离线评测覆盖不到的边缘场景**。离线评测集再好，也覆盖不了生产环境的全部多样性。回放测试自动把真实场景补充进来。

**量化"线上表现"**。在评测集上跑的 TSR 是在"限定场景"下的 TSR。回放测试的 TSR 更接近"真实 TSR"。

**检测 concurrency 问题**。生产环境有多个 Agent 实例并发处理请求，某些问题只在特定并发条件下出现。回放时可以模拟这种压力。

## 在线监控与漂移检测

### 什么是漂移

Agent 的"漂移"（drift）指它的行为随时间发生非预期的变化。

漂移的来源：

| 漂移类型 | 来源 | 表现 |
|---------|------|------|
| 模型漂移 | LLM 模型升级/降级 | 同一 prompt 下输出质量变化 |
| 数据漂移 | 用户输入模式变化 | Agent 遇到没见过的输入类型 |
| 工具漂移 | 外部 API 变化 | 工具返回格式或行为变化 |
| prompt 漂移 | 多次 prompt 修改累积 | 逐步偏离原始设计目标 |

### 漂移检测方法

**TSR 趋势监控**：

```
每日 TSR → 7 日移动平均线 → 14 日移动平均线

告警条件:
  - 单日 TSR 较 7 日均值下降 > 5%
  - 7 日均线持续下降 3 天
  - TSR 绝对值低于 85%
```

**输入分布监控**：

```
监控用户输入的主题分布变化。
如果"退款申请"类输入突然从 5% 涨到 20%，
Agent 对这类输入的处理能力可能不足。

方法: 用分类器或聚类对用户输入做主题分类，监控分布变化。
```

**输出质量监控**：

```
对生产环境的 Agent 输出做随机抽样（比如 1%）：
  → 自动 LLM-as-Judge 评分
  → 如果评分均值下降 > 阈值 → 告警

注意: 生产环境做 LLM-as-Judge 要考虑成本和延迟。
建议抽样 + 离线批量处理，不在请求链路中实时执行。
```

## 评测与可观测性的数据打通

评测和可观测性（第 13 章）需要打通数据：

### Trace 关联评测结果

```
Trace: req_abc123
  ├── user 输入: "帮查订单"
  ├── Agent 行为: query_order → response
  ├── 离评论证: 通过（100% 匹配）
  ├── 用户反馈: 👍（好评）
  └── 在线评分: 4.8/5 (LLM-as-Judge 在线抽样)
```

一条 Trace 同时关联了离线结果和在线反馈。当用户投诉时：

```
用户投诉 → 找到 Trace ID → 查看：
  - 离线评测这个用例通过了没有？
    如果通过了 → 评测集不够全面，需要补充
    如果没通过 → 为什么没在上线前发现？
  - 用户反馈了什么？
  - 在线评分多少？
```

### 评测集回填流程

生产环境中最有价值的信号应该定期回填到评测集中：

```
生产日志
  ├── 用户投诉 → 人工标注 → 加入对抗评测集
  ├── 用户差评 → 自动聚类 → 抽代表性用例 → 加入评测集
  ├── Agent 出错 → 自动提取 → 加入回归测试集
  └── 漂移检测异常 → 分析根因 → 补充评测集
```

这样评测集才能"保鲜"——始终保持对真实场景的覆盖。

## 生产评测的落地路径

### 分阶段搭建

```
第一阶段: 离线评测体系
  └── 离线评测集 + CI/CD 门禁
  └── LLM-as-Judge 自动化评分
  └── 确立基线指标

第二阶段: 在线信号接入
  └── 用户显式反馈 (👍/👎)
  └── 用户隐式信号采集
  └── Trace 关联评测结果

第三阶段: 回放与漂移
  └── 生产回放评测
  └── TSR 趋势监控
  └── 输入分布漂移检测

第四阶段: 闭环优化
  └── 生产信号自动回填评测集
  └── 离线+在线联合分析
  └── 自动告警 + 建议修复方向
```

大多数团队从第一阶段开始，逐步推进。**第一阶段做完就已经比 90% 的团队强了**。

## 总结

生产环境评测回答离线评测回答不了的问题——"用户真的觉得好用吗"和"系统在变差吗"。

核心要点：离线保证"上线前"，在线保证"上线后"；用户反馈（显式+隐式）是最真实的信号；回放测试将生产数据反哺离线评测；评测与可观测性打通数据，形成闭环。
