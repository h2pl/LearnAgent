## 用 Trace 找到性能瓶颈

### 标准分析流程

当用户反馈"Agent 有点慢"时，按这个流程定位：

<p align="center">
  <img src="../../assets/14-observability/perf-analysis-flow.svg" alt="性能瓶颈定位四步流程" width="95%"/>
</p>

**第一步：确认是普遍问题还是个别问题**

看 P50 / P95 / P99 延迟：

```
如果 P50 正常、P99 很高 → 个别慢请求的问题（优化尾部延迟）
如果 P50 也高了 → 普遍变慢（找到所有请求的共同原因）
```

**第二步：找到最慢的 Span**

对一批慢请求，按 Span 类型聚合耗时：

```
查询: 过去 1 小时内 P95 > 5s 的请求
按 Span type 聚合总耗时占比:

llm_call:       68% ← 最大瓶颈
tool_call:      18%
retrieval:      10%
framework:       4%
```

**第三步：下钻到具体的慢 Span**

找出占大头的那类 Span，再看具体是哪个操作慢：

```
llm_call 中:
  gpt-4o:  平均 2.1s  ← 占 LLM 时间大头
  gpt-4o-mini: 平均 0.8s

tool_call 中:
  query_order:  平均 450ms
  send_email:   平均 120ms
  external_api: 平均 1.8s  ← 这里慢了
```

**第四步：优化影响最大的项**

不要优化占比小的。把精力花在影响最大的瓶颈上。如果 LLM 调用占 68% 总时间，优化 tool_call 中那几百毫秒毫无意义。

### Trace 能直接告诉你的事

有了结构化的 Trace 数据，很多问题不需要"猜测"：

```
"过去一周 P95 延迟升了 40%，是不是换模型了？"
  → 查 Trace 数据，对比两周的 llm_call 耗时分布
  → 发现 gpt-4o-mini 占比从 60% 降到了 30%
  → 根因：模型路由规则被意外修改
```

Trace 数据让你能回答"同比之前差了多少、差在哪个环节"。这是日志做不到的——日志告诉你"出错了"，Trace 告诉你"在哪一步、为什么、花了多久"。

## Agent 特有的延迟模型

Agent 系统的延迟和传统 API 完全不同——它不是一次请求-响应，而是**多轮循环**。理解这个延迟模型是性能分析的基础。

### 延迟构成

```
一个 Agent 请求的总延迟 =
  多次 LLM 调用的串行延迟
+ 每次 LLM 调用的等待时间（TTFT + 生成时间）
+ 工具调用的网络延迟
+ 检索延迟
+ 框架开销
```

关键在于**串行**——Agent 通常是一轮一轮执行的，下一轮依赖上一轮的输出。5 轮循环，每轮 2s，总延迟就是 10s，没法并行。所以优化 Agent 延迟的核心是：**减少轮次、缩短每轮耗时**。

### 五种常见延迟模式

**模式一：LLM 首 Token 等待（TTFT 高）**

现象：LLM Span 很长，但绝大部分时间是 TTFT（首 token 生成时间）。

根因：模型太大、prompt 太长、并发太高导致模型服务端排队。

缓解：换更快的模型、减少输入 token、水平扩展推理节点。

**模式二：上下文窗口膨胀**

现象：越往后 Step，LLM 的输入 token 越来越多，耗时越来越长。

```
Step 1: 1200 tokens → 0.8s
Step 2: 2400 tokens → 1.5s
Step 3: 4800 tokens → 2.8s  ← 越来越慢
Step 4: 7200 tokens → 4.2s
```

这种模式在 Trace 中非常容易识别——同类型 Span 的 latency 递增，且 input_tokens 属性同步递增。

根因：每轮把全部对话历史塞进 prompt，没有做压缩或裁剪。

缓解：对话历史压缩、窗口滑动、用摘要替换原始历史。

**模式三：工具调用链过深**

现象：Agent 为了完成一个简单任务，调用了 5-6 个工具。

根因：Agent 的决策策略有问题，把单步能完成的事情拆成了多步。也可能是工具粒度太细——`get_user` + `get_address` + `get_orders` 三个工具可以合并为一个 `get_user_profile`。

缓解：重构工具粒度、在 prompt 中引导"尽量一步完成"、限制最大工具调用次数。

**模式四：死循环**

现象：同一种工具被反复调用，每次参数略有不同。Trace 中会出现一串相同 type 的 Span。

```
tool_call:search_kb (q="退款政策")
tool_call:search_kb (q="退款流程")
tool_call:search_kb (q="退款条件")
tool_call:search_kb (q="退款时限")
```

根因：Agent 在某个状态卡住了，既没有成功也没有失败，也不知道什么时候该停。常见于检索 RAG 场景——第一次没找到满意答案就换关键词再查。

缓解：设置最大步数硬上限、检测重复模式自动终止、在 prompt 中加入"三次找不到就告诉用户查不到"的指令。

**模式五：外部依赖慢**

现象：工具调用的 Span 中，大部分时间花在等待外部 API 返回。Trace 上表现为 tool_call Span 的 duration 远高于预期，且 status 没有报错。

根因：外部服务慢或不可靠，或者超时设置不合理导致本应快速失败的重试消耗了额外时间。

缓解：加缓存（相同参数的结果缓存）、设合理的超时（不要让一个失败请求等 30 秒）、考虑异步（非实时场景走消息队列）。

### 延迟排查清单

当用户反馈"Agent 慢"时，按这个清单快速定位：

```
□ 整体 P95/P99 是否显著变化？→ 是全局问题还是个别问题
□ 哪类 Span 占总延迟比例最高？→ 找最大瓶颈
□ LLM 的 TTFT 是否异常高？→ 模型太大或 prompt 太长
□ 输入 token 是否每轮都在膨胀？→ 上下文没有做压缩
□ 工具调用链是否过深？→ 工具粒度或决策策略问题
□ 是否有工具在反复调用？→ 死循环
□ 外部 API 是否有超时或重试？→ 外部依赖不可靠
```

## 分层优化策略

找到瓶颈后，按层优化。每层的优化手段不同。

### 模型层

```
优化手段                        延迟影响    实现难度
──────────────────────────────────────────────────
更快的模型                      -20~50%     低（换 endpoint）
减少输入 token                  -10~40%     中（改 prompt 策略）
KV Cache / prompt 缓存         -30~50%     中（需缓存基础设施）
量化推理                        -20~40%     高（需自部署）
水平扩展推理节点                -30~70%     高（需推理集群）
```

减少输入 token 的效果往往被低估。一个 Agent 请求的 LLM 输入 token 可能从 1K 膨胀到 8K+，而 LLM 的延迟与输入 token 数大致成正比。**压缩 prompt 是最免费的加速手段**。

### Prompt 层

```
优化手段                        延迟影响    成本影响
──────────────────────────────────────────────────
精简 system prompt              -5~10%     -10%
对话历史压缩（限 N 轮+摘要）    -20~40%    -20%
检索内容裁剪（只保留最相关段）  -10~20%    -15%
```

### 缓存层

```
缓存层次         缓存对象               典型命中率  加速效果
─────────────────────────────────────────────────────────
Prompt KV Cache  相同 system prompt     50-70%    减少 TTFT
检索结果缓存     相同 query 的检索结果   40-60%    避免检索耗时
工具结果缓存     相同 API 调用结果       20-40%    避免外部调用
```

注意：不要对 LLM 响应做缓存。同样的用户输入在不同上下文下，Agent 应该给出不同回答。缓存会导致"机器人感"。

### 架构层

```
优化手段                        延迟影响    实现难度
──────────────────────────────────────────────────
规则前置（if 解决的不交给 LLM）  -90%+      低
有限状态机替代 LLM 循环          -80%+      中
异步非阻塞                      并发能力↑   中
批量推理（非实时场景）           吞吐量↑   中
```

架构层优化是最彻底的，但也是实现成本最高的。**规则前置是最被低估的优化**——能用 `if` 语句判断的，不要交给 LLM。一个 `if` 是微秒级的，一次 LLM 调用是秒级的。一个典型模式：

```
用户输入 → 规则匹配（正则 / 关键词）
  ├── 匹配已知模式 → 直接返回预制回答（微秒级）
  └── 未知模式 → 交给 LLM 处理（秒级）
```

对于客服场景，20-30% 的查询可以走规则通道。这部分请求的延迟直接从秒级降到毫秒级。

## 案例：一次完整的性能优化

### 背景

客服 Agent，日活 10 万用户，日均 30 万请求。

### 现状测量

从 Trace 数据中提取基线：

```
每请求平均延迟: 4.2s (P95)
每请求平均 LLM 调用次数: 3.5

延迟构成:
  LLM 调用: 72%
  工具调用: 18%
  检索: 8%
  框架: 2%
```

LLM 调用占 72%——优化重点明确。

### 瓶颈识别

分析 Trace 数据发现两个性能问题：

**问题 1**：所有请求都用 gpt-4o。但 Trace 数据显示 60% 的请求平均输入 token 不到 500、不需要复杂推理——用 gpt-4o-mini 就可以。

**问题 2**：Agent 的对话历史不做裁剪，每轮 Step 都将全部历史塞入上下文。Trace 中看到 input_tokens 属性从 Step 1 的 1,200 增长到 Step 3 的 4,800，LLM 耗时同步从 0.8s 增长到 2.8s。

**问题 3**：外部 API 超时 5s，重试次数 3 次。Trace 中看到一个失败的工具调用因为重试消耗了 15 秒。

### 优化方案

```
优化 1: 模型路由
  简单请求 → gpt-4o-mini (60% 流量)
  复杂请求 → gpt-4o (40% 流量)
  预计: 延迟 -40%

优化 2: 对话历史压缩
  限制最大历史轮数 = 6 轮
  超过 6 轮的历史做摘要
  预计: 延迟 -20%

优化 3: 外部 API 超时 + 重试调整
  超时从 5s → 2s
  重试次数从 3 → 1
  预计: 尾部延迟 -30%
```

### 效果量化

优化后重新测量：

```
每请求平均延迟: 1.8s (P95)    ↓ 57%
每请求平均 LLM 调用: 2.0      ↓ 43%
```

## 总结

性能分析的核心是用 Trace 数据回答"慢在哪"——**不猜测、不靠感觉**。

四步流程：看整体延迟分布（是 P50 高还是 P99 高）→ 按 Span 类型聚合找最大占比 → 下钻到具体操作 → 优化影响最大的项。

Agent 特有的延迟模型（多轮串行）决定了优化重点和传统 API 不同——控制上下文大小、减少 LLM 调用轮次、规则前置——这些手段的影响远大于单纯换一个更快的模型。
