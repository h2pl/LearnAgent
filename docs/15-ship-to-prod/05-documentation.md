> 没有文档的系统是一个"只有你会修"的系统。文档让知识可传承、决策可追溯、新人可上手。Agent 系统的文档体系和传统项目稍有不同——多了 prompt 版本管理、工具描述文档、评测集说明。

你好，我是江小湖。这是产品交付章的最后一篇。前面四篇文章覆盖了架构、API、部署、运维——让系统跑起来了。但还有一个重要问题：**你休假的时候，系统出了故障，值班的人能搞定吗？**

项目文档回答的就是这个问题。

## 项目文档体系

一个 Agent 项目的文档体系应该覆盖两个维度的需求——**运维需求**（系统出问题了怎么修）和**开发需求**（新成员怎么上手、下次迭代怎么改）。

```
docs/
├── README.md              # 项目概述 —— 新人第一件事
├── ARCHITECTURE.md         # 架构设计 —— 系统全局视图
├── API.md                  # API 文档 —— 外部集成指南
├── DEPLOYMENT.md           # 部署文档 —— 如何上线
├── EVALUATION.md           # 评测说明 —— 怎么评测、评测集在哪
├── PROMPTS.md              # Prompt 工程 —— 所有 prompt 模板
├── SECURITY.md             # 安全策略 —— 安全配置与基线
├── RUNBOOK.md              # 运维手册 —— 出问题了怎么办
├── CHANGELOG.md            # 变更日志 —— 什么版本改了啥
└── adr/                    # 架构决策记录
    ├── ADR-001-langgraph-framework.md
    └── ADR-002-vector-db-choice.md
```

### README

回答四个问题，每个问题一句话：

```
# Agent 订单助手

这是一个基于 LLM 的订单管理助手，支持用户查询订单状态、修改地址、取消订单。

## 核心能力
- 查询订单状态和物流信息
- 修改订单收货地址
- 取消未发货的订单

## 快速开始
```bash
cp .env.example .env
docker compose up -d
curl http://localhost:8000/health
```

## 技术栈
Python 3.12 + FastAPI + LangGraph + PostgreSQL + Redis + Qdrant
```

README 不写实现细节。调用 API 的详细说明在 `API.md`，架构细节在 `ARCHITECTURE.md`。README 只需要让一个人**在 5 分钟内决定要不要看下去**。

### ARCHITECTURE

包含系统架构图、数据流图、关键设计决策的总结、已知限制和技术债。例如：

```
# 架构设计

## 系统架构
[架构图链接]

## 数据流
用户请求 → API 网关 → Agent 引擎 → LLM / 工具 → 响应

## 设计决策
核心设计决策详见 adr/ 目录。
关键决策摘要：
1. 使用 LangGraph 而不是自研引擎 — 减少重复造轮（ADR-001）
2. 使用 Qdrant 作为向量数据库 — 性能最优，社区活跃（ADR-002）
3. 使用 SSE 而不是 WebSocket — 满足当前需求，实现更简单

## 已知限制
- 不支持语音输入
- 单次请求最大执行 30 秒
- 不支持流式中断恢复（断线后需要重新发送请求）
```

## 架构决策记录 (ADR)

ADR 是本文档体系中最重要的部分——它记录了**"为什么我们这么做"**。没有 ADR，一个设计决策的理由会随着团队成员的离开而丢失。

### ADR 格式

```markdown
# ADR-001: 使用 LangGraph 作为 Agent 编排框架

## 状态
已接受 (2026-05-01)

## 背景
需要选择一个 Agent 编排框架来驱动 Agent 的执行循环。
候选方案: LangGraph、CrewAI、自研引擎。

## 决策
选择 LangGraph 作为核心编排框架。

## 理由
1. 社区最活跃，文档完善，遇到问题容易找到解决方案
2. 原生支持状态图（StateGraph），适合复杂的多步 Agent 流程
3. 内置 Tracing 支持（LangSmith），无需额外开发可观测性
4. 团队中 2 人已有 LangChain 使用经验，学习成本低

## 替代方案
- CrewAI：固定团队结构，动态任务路由支持不足
- 自研引擎：灵活度最高，但开发成本太大（预估 2 人月）

## 影响
正面：
- 加速开发（预计节省 2 人月的开发时间）
负面：
- 对框架产生依赖，版本升级可能带来兼容性问题
```

### ADR 的状态流转

```
提议中 → 已接受 → 已过时（被新 ADR 取代）
       → 已拒绝
```

ADR 被新决策取代时，不是删除旧的，而是标记为"已过时"并在新 ADR 中引用旧 ADR：

```
ADR-003: 将编排框架从 LangGraph 迁移到自研引擎
替换: ADR-001
理由: LangGraph 的版本兼容问题导致频繁 blocking，团队已有足够经验自研
```

### 什么时候需要写 ADR

```
需要写:
  - 技术选型（框架、数据库、LLM 模型）
  - 架构变更（分层方式、通信协议）
  - 接口设计（API 风格、数据格式）
  - 安全策略（认证方式、权限模型）

不需要写:
  - 日常 bug 修复
  - 小的配置变更
  - 工具新增/修改
```

ADR 不是用来记录"每一行代码怎么写的"，而是记录"重要的、不可逆的、有长期影响的设计决策"。

## Agent 专属文档

Agent 系统有一些传统项目没有的文档需求。

### Prompt 版本管理

Prompt 是 Agent 系统的"代码"之一。每个 prompt 模板应该有独立的版本记录：

```markdown
# Prompt 模板: order-agent-system

## 当前版本 v4 (2026-06-18)
修改: 添加了"取消已发货订单"的处理逻辑
位置: prompts/order-agent-system/v4.txt
TSR 影响: 取消订单 TSR 从 87% → 94%

## 历史版本
| 版本 | 日期 | 修改 | TSR 变化 |
|------|------|------|---------|
| v4 | 06-18 | 添加已发货订单取消逻辑 | +7% |
| v3 | 06-10 | 精简 system prompt | +2% |
| v2 | 06-01 | 添加工具调用示例 | -1% |
| v1 | 05-20 | 初始版本 | — |
```

Prompt 变更影响大、回滚困难。版本记录让你能精确回答"换了这个 prompt 后 TSR 变了多少"。

### 工具描述文档

工具的描述（决定 LLM 能不能选中正确的工具）和工具的实现代码一样重要：

```markdown
# 工具: query_order

## 描述
根据订单 ID 查询订单的当前状态和详细信息。
LLM 应该在这个用户场景下调用此工具：
- "查一下我的订单到哪里了"
- "ORD-123 的状态是什么"
- "我的订单什么时候发货"

## 参数
| 参数 | 类型 | 必需 | 说明 |
|------|------|------|------|
| order_id | string | 是 | 订单 ID，格式 ORD-{数字} |

## 返回
| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 订单 ID |
| status | string | pending/shipped/delivered/cancelled |
| eta | string | 预计送达日期 |
```

### 评测集说明

评测集也需要文档——不仅仅是数据，还包括它的设计思路：

```markdown
# 评测集: order-agent-eval

## 规模
- 黄金集: 50 条（核心流程）
- 回归集: 200 条（从生产日志中标注）
- 对抗集: 30 条（已知弱点针对性测试）
- 空跑集: 10 条（CI/CD 门禁）

## 数据分布
- 查询订单: 40%
- 修改地址: 25%
- 取消订单: 20%
- 退款申请: 15%

## 标注标准
- 完全成功 (1.0): 用户目标完美完成，路径最优
- 部分成功 (0.5): 主要目标达成，但有细节缺失
- 失败 (0.0): 主要目标未达成

## 更新记录
| 日期 | 变更 | 原因 |
|------|------|------|
| 06-18 | 新增 10 条退款用例 | 用户反馈退款场景覆盖不足 |
| 06-10 | 更新 5 条地址修改标注 | 标注标准不一致，重新标注 |
```

## 变更日志 (Changelog)

CHANGELOG 给用户和同事看的，不是给 Git 看的。Git log 是给机器看的。

```markdown
# Changelog

## [1.3.0] - 2026-06-18
### 新增
- 支持查询历史订单（之前只能查最近一笔）
- 流式响应现在支持取消操作（用户可中断正在执行的 Agent）

### 修复
- 修复 cancel_order 在订单已发货时返回的报错信息不友好 (#123)
- 修复 SSE 连接在空闲 30 秒后自动断开的问题 (#118)

### 变更
- prompt 模板升级到 v4（取消订单场景 TSR +7%）

### 依赖
- LangGraph 从 0.2.0 升级到 0.3.0（修复状态序列化 bug）

## [1.2.0] - 2026-06-01
### 新增
- 新增流式响应（SSE）支持
- 新增用户反馈收集（👍/👎 按钮）
```

Changelog 的受众是**不熟悉代码库的人**——产品经理、运营、新加入的工程师。不要写"修复了 create_order 的参数校验 bug"，要写"修复了创建订单时地址格式校验不通过的 bug"。

## 文档的维护

文档和代码一样需要维护。一个过时的文档比没有文档更糟糕——它会误导人。

### 保持文档鲜活的策略

```
代码审查中包含文档审查:
  PR 添加了新工具 → 需要同时更新 tools.md 和工具描述文档
  PR 修改了 API → 需要同时更新 API.md
  PR 做了架构变更 → 需要写 ADR

CI 中检查文档:
  - 检测 .md 文件的最后修改时间是否和对应的代码文件相差太大
  - API 文档和 OpenAPI spec 是否同步

定期文档清理:
  - 每季度评审所有 ADR，标记已过时的
  - 检查 README 中的"快速开始"步骤是否还能跑通
```

### 没人读的文档不如不写

写文档前先问：**谁会读这个文档？他们什么时候读？读了之后做什么？**

```
如果有人问"系统怎么部署" → DEPLOYMENT.md
如果有人问"API 怎么调" → API.md
如果有人问"为什么选这个数据库" → ADR-002
```

如果一个问题从没有人问过，就不要为它写文档。

## 总结

项目文档是"知识传承"的基础设施。Agent 系统的文档比传统项目多了 prompt 版本管理、工具描述、评测集说明三个专属领域。

五个核心文档：README（5 分钟上手）→ ARCHITECTURE（系统全局视图）→ ADR（设计决策记录）→ CHANGELOG（变更记录）→ RUNBOOK（故障处理）。外加 Agent 专属的 prompt 版本记录、工具描述文档、评测集说明。

本系列 15 章从 LLM 基础讲到了交付运维，覆盖了构建一个生产级 Agent 系统所需的全部知识。最后一句想说的是：**知识不是学完的，是用完的。动手写一个 Agent，让它跑起来。**
