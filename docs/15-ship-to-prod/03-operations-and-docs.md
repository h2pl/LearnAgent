## 日常巡检

Agent 系统的日常巡检与传统后端服务不同——除了基础设施健康，还要关注"逻辑健康"。

### 巡检清单

```
每日巡检:
  ├── [ ] 任务完成率 vs 前一天
  ├── [ ] P95 延迟是否在基线内
  ├── [ ] 错误率是否异常
  ├── [ ] 外部依赖（LLM API、工具 API）是否正常
  └── [ ] 关键指标的 7 日趋势

每周巡检:
  ├── [ ] Token 消耗趋势，是否有异常增长
  ├── [ ] 预算消耗进度
  ├── [ ] 用户满意度评分变化
  ├── [ ] 评测集指标 vs 基线
  └── [ ] 安全审计日志检查

每月巡检:
  ├── [ ] 性能容量评估（是否需要扩容）
  ├── [ ] 依赖版本更新检查
  ├── [ ] 密钥轮换
  └── [ ] Runbook 演练
```

### 巡检自动化

巡检的大部分步骤应该自动化，每日巡检甚至可以全自动。只有需要人工判断的项（如用户满意度趋势分析）才留给人。

```python
# 示例：自动巡检脚本
def daily_check():
    reports = []
    reports.append(check_task_completion_rate())
    reports.append(check_p95_latency())
    reports.append(check_error_rate())
    reports.append(check_external_dependencies())
    
    if any(r.is_abnormal for r in reports):
        send_alert("每日巡检发现异常", reports)
    else:
        log_info("每日巡检全部通过")
```

## 故障响应

### 告警分级

| 级别 | 定义 | 响应 | 通知方式 |
|------|------|------|---------|
| P0 | 服务不可用 / 核心功能崩溃 | 立即 | 电话 + 即时消息 |
| P1 | 核心功能受损 / 错误率飙升 | 15 分钟 | 即时消息 |
| P2 | 非核心功能异常 | 2 小时 | 邮件 |
| P3 | 预警信息 | 24 小时 | 仪表盘 |

### 故障响应流程

```
1. 感知: 告警触发或用户投诉
   ↓
2. 确定影响范围: 
   - 哪些用户受影响？
   - 哪个功能出问题？
   - 是否涉及数据安全？
   ↓
3. 定位根因:
   - 查看 Trace 定位失败环节
   - 检查最近变更
   - 查看依赖服务状态
   ↓
4. 止血:
   - 回滚最近部署（优先）
   - 降级，切换到备用模型/工具
   - 限制流量，防止影响扩散
   ↓
5. 恢复:
   - 执行修复操作
   - 验证功能正常
   ↓
6. 复盘:
   - 根因分析文档
   - 改进措施
   - 更新 Runbook
```

### 止血优先原则

**故障发生时，先止血，后找根因。**

```
错误做法: 花 30 分钟定位"为什么 GPT-4o 返回了错误格式"
正确做法: 立即切换到 gpt-4o-mini 或备用模型，恢复服务后再分析根因

错误做法: 花 1 小时排查"为什么 Redis 连接失败"
正确做法: 立即重启 Redis 或切换到备用 Redis 实例
```

止血就是**以最快速度恢复到用户可用状态**，不管"为什么"。

## 灾难恢复

### 数据备份

| 数据类型 | 备份频率 | 保留策略 | 恢复方式 |
|---------|---------|---------|---------|
| 对话历史 | 实时归档 | 30 天 | 即席查询 |
| 向量数据库 | 每日全量 | 7 天 | 全量恢复 |
| 关系型数据库 | 每日全量 + WAL | 30 天 | PITR 恢复 |
| 配置文件 | Git 版本控制 | 永久 | Git checkout |
| 模型配置 (prompt) | Git 版本控制 | 永久 | Git checkout |

### 备份验证

**没有验证过的备份等于没有备份。** 每月至少做一次恢复演练：

```bash
# 恢复演练脚本
1. 从备份恢复向量数据库到临时实例
2. 验证恢复后的数据完整性（文档数、索引状态）
3. 运行 5 个核心评测用例验证
4. 清理临时实例
5. 生成恢复演练报告
```

### 恢复流程

```
灾难场景: 向量数据库完全损坏

恢复步骤:
1. 创建新的 Qdrant 实例
2. 从最近备份恢复数据
3. 验证数据完整性（文档数匹配）
4. 切换 Agent 配置指向新实例
5. 监控错误率，确认恢复成功
6. 如果备份也损坏 → 从原始文档重新构建索引
```

## 运维手册

每个 Agent 系统都应该有一份运维手册，至少包含：

```markdown
RUNBOOK.md
├── 系统架构图
├── 服务依赖关系（含外部 API 清单）
├── 启动/停止/重启步骤
├── 常见故障排查流程
│   ├── LLM 调用失败
│   ├── 工具调用失败
│   ├── 高延迟
│   └── 任务完成率下降
├── 备份与恢复步骤
├── 扩容流程
├── 每周巡检清单
└── 紧急联系人
```

Runbook 的关键是**可执行性**——一个不熟悉系统的人照着 Runbook 也能恢复服务。

## 技术文档

### 架构决策记录 (ADR)

```markdown
# ADR-001：使用 LangGraph 作为 Agent 编排框架

## 背景
需要选择 Agent 执行引擎

## 决策
使用 LangGraph 作为核心编排框架

## 理由
1. 社区最活跃，文档最完善
2. 原生支持状态图、条件分支
3. 团队已有使用经验

## 替代方案
- CrewAI：固定团队结构，动态任务支持不足
- 自研引擎：开发成本太大

## 影响
正面：加速开发，效率提升
负面：框架依赖，升级需关注 breaking changes
```

ADR 最大的价值是记录"我们为什么这么做"。
