你好，我是江小湖。前面三篇文章覆盖了评测和追踪——怎么测量 Agent 的质量、怎么看到它的内部状态。但还有一个不可忽视的维度：**成本**。

一个生产环境的 Agent，每天处理 10 万请求、平均每次调用 3 次 LLM、每次 LLM 调用消耗 2000 token——一个月就是 18 亿 token。按月账单算下来，可能是五位数甚至六位数人民币。**没有监控的成本，迟早会让你惊讶**。

## Agent 成本模型

Agent 的成本比传统 API 调用复杂得多。一个请求可能触发多次 LLM 调用、多次工具调用、多次检索。

### 成本构成

<p align="center">
  <img src="../../assets/12-eval-trace/cost-breakdown.svg" alt="Agent 成本构成" width="95%"/>
</p>

| 成本项 | 典型占比 | 影响因素 |
|--------|---------|---------|
| LLM 推理 | 70-85% | 模型价格、token 量、调用次数 |
| 向量检索 | 5-15% | 文档量、检索频次、索引类型 |
| 工具调用 | 3-10% | 外部 API 单价、调用频次 |
| 基础设施 | 2-5% | 服务器、存储、网络 |

**LLM 推理成本是绝对大头**，优化 Agent 成本的核心就是优化 LLM 调用的 token 消耗。

### 成本公式

```
单次 Agent 请求成本 =
  Σ(每次 LLM 调用 (input_tokens × input_price + output_tokens × output_price))
  + Σ(每次检索 (检索计费单位 × 单价))
  + Σ(每次工具调用 (API 价格))
```

由于 LLM 占据绝对多数，简化公式可以近似为：

```
成本 ≈ 请求量 × 平均每请求 token 数 × 模型单价
```

## 成本核算体系

### 核心指标

| 指标 | 定义 | 计算公式 |
|------|------|---------|
| 每次请求平均成本 | 一次 Agent 执行的总成本 | 总成本 / 请求数 |
| 每次 LLM 调用平均 token | 单次 LLM 的 token 消耗 | 总 token / LLM 调用次数 |
| 每请求 LLM 调用次数 | 平均触发几次 LLM | LLM 调用次数 / 请求数 |
| 输入输出比 | 输入 token 与输出 token 的比例 | input_tokens / output_tokens |
| 缓存命中率 | prompt 缓存减少的 token | 缓存节省 token / 总 token |

### 按维度聚合

成本需要从多个维度聚合分析才能发现优化机会：

**按模型聚合**：gpt-4o 花了多少钱？gpt-4o-mini 花了多少钱？各占比多少？

**按用户聚合**：哪些用户是"token 消耗大户"？他们的使用模式合理吗？

**按功能聚合**：功能 A 和功能 B，哪个成本更高？哪个 ROI 更高？

**按时间聚合**：每天的 token 消耗趋势如何？周末和上班日的成本差异大吗？

## 性能指标体系

成本不能孤立看，必须结合性能指标才有意义。

### 关键性能指标

| 指标 | 基准值 | 告警阈值 |
|------|--------|---------|
| P50 响应时间 | < 2s | > 5s |
| P95 响应时间 | < 5s | > 10s |
| P99 响应时间 | < 10s | > 20s |
| 首次 token 时间 (TTFT) | < 500ms | > 2s |
| 错误率 | < 1% | > 5% |
| 重试率 | < 2% | > 10% |

### 成本-性能关联分析

**成本-性能矩阵**是决策的核心工具：

```
             低成本         高成本
               │              │
  高性能    ┌───── Good ── Trade-off
  低性能    │  Review ────  Bad
```

- **Good 象限**（高性能 + 低成本）：理想状态，保持
- **Trade-off 象限**（高性能 + 高成本）：可能用了过于昂贵的方案
- **Review 象限**（低性能 + 低成本）：可能在降本上过了头
- **Bad 象限**（低性能 + 高成本）：系统有大问题，需要紧急优化

**每次成本优化，都要检查是否把某个指标推到了 Bad 象限**。

## 优化策略

成本优化应遵循**先定指标再优化**的原则——没有明确指标的情况下谈降本，可能损伤用户体验。

### 模型路由

不是所有请求都需要最强模型。用**分类器 + 模型路由**做成本分级：

```
用户输入
  │
  ▼
分类器（规则 or 小型 LLM）
  ├── 简单任务 → gpt-4o-mini (成本降低 80-90%)
  ├── 复杂任务 → gpt-4o (需要全面推理能力)
  └── 特殊任务 → o4-mini (需要复杂推理链)
```

实践中，**60-70% 的请求可以用小型模型处理**，而用户感知不到差异。

### Prompt 压缩

LLM 的输入 token 往往是输出的好几倍。压缩 prompt 能显著降低成本：

- **精简 system prompt**：去掉冗余示例，每个字符都只为达到目标服务
- **对话历史压缩**：用摘要代替原始对话历史，或者只保留最近 N 轮
- **检索压缩**：检索内容只保留最相关的片段而不是整篇文档
- **结构化提示**：用 JSON/XML 替代自然语言描述，减少 token

### 缓存策略

缓存是降本最直接的手段。多层缓存策略：

| 缓存层次 | 缓存对象 | 典型命中率 | 效果 |
|---------|---------|-----------|------|
| Prompt 层 | 相同 system prompt 的 KV Cache | 50-70% | 减少 TTFT |
| 响应层 | 相同输入对相同输出 | 10-30% | 完全避免 LLM 调用 |
| 检索层 | 相同 query 的检索结果 | 40-60% | 避免重复检索 |
| 工具层 | 相同 API 调用的结果 | 20-40% | 避免重复外部调用 |

### 批量处理

对于延迟不敏感的任务，用批量 API 替代实时 API：

```
实时 API：每条 1 次调用，100 条 = 100 次调用
批量 API：1 次调用包含 100 条，共享 prompt 前缀
成本节省：40-60%
```

适合批量处理的场景：内容审核、批量分类、数据标注、报告生成。

### 减少 LLM 调用次数

这是最根本的优化。**每减少一次 LLM 调用，就省掉 100% 的那次调用成本**。

- **规则前置**：能用规则判断的，不交给 LLM。例如简单意图识别用关键词匹配
- **一次完成多件事**：合并多个简短 LLM 调用为一个复杂度更高的调用
- **有限状态机**：对于确定性流程，用有限状态机代替 LLM 循环

## 预算管控

### 预算分配模型

按层级分配预算，每一层设置上限：

```
月度总预算
├── 模型 A 预算上限 (gpt-4o)
├── 模型 B 预算上限 (gpt-4o-mini)
├── 向量检索预算上限
└── 外部工具预算上限
```

### 预算保护机制

**硬限制 vs 软限制**：

- **软限制**：超过 80% 预算时发送告警，服务自动使用更经济的模型
- **硬限制**：超过 100% 预算时，非核心功能降级或停止服务

**每个请求的预算**：

为每个请求设置推理预算上限——如"单个请求最多调用 10 次 LLM"或"单个请求最多消耗 50,000 token"。超过上限的请求自动终止并返回兜底回复。

### Token 预留

对于高优先级用户（VIP、付费用户），可以为他们在高成本模型上预留 token 配额。确保高价值用户不会因为通用预算用尽而体验降级。

## 监控告警

一套完整的成本监控系统应该覆盖以下告警维度：

| 告警类型 | 条件 | 响应 |
|---------|------|------|
| 预算预警 | 日消耗 > 日均预算 × 1.5 | 检查是否有异常流量 |
| 成本飙升 | 成本月环比增长 > 20% | 分析驱动因素 |
| Token 异常 | 单请求 token > 平均值 × 5 | 检查是否有死循环或过大上下文 |
| 模型偏斜 | 高成本模型占比 > 预期值 | 检查模型路由是否正常工作 |
| 性能退化 | 在成本不变时 P95 延迟 > 基准 2x | 排查基础设施问题 |

## 实践案例

### 场景：客服 Agent

**状态**：日活 10 万用户，日均 30 万请求

**优化前**：所有请求调用 gpt-4o，平均每请求 3 次 LLM 调用，月成本约 ¥120,000

**优化后**：

1. **模型路由**：60% 简单请求 → gpt-4o-mini，节省 50% token 成本
2. **缓存**：常见问答缓存命中率 25%，减少 25% 的 LLM 调用
3. **Prompt 压缩**：对话历史从 20 轮压缩到 8 轮 + 摘要，减少 40% 输入 token
4. **死循环检测**：同一工具连续调用 > 3 次时自动终止，减少异常消耗

**结果**：

- 月成本：¥120,000 → ¥43,000（节省 64%）
- P95 延迟：4.2s → 2.1s（更快的模型处理简单请求）
- 用户满意度：不变（用户没察觉到变化）

## 总结

成本优化不是"省钱比赛"，而是在**成本、性能和用户体验之间找到平衡点**。

核心原则：先测量（每个请求的 token 和成本）→ 再设置基准（正常消耗是多少）→ 再优化（从影响最大的项开始）→ 持续监控（防止退化）。
