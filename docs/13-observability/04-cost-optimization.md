你好，我是江小湖。上一篇文章讲性能——怎么让 Agent 更快。本文讲另一半——**怎么让 Agent 更便宜**。

性能和成本是同一枚硬币的两面——同一个 Trace 数据既回答"哪里慢"也回答"哪里贵"。但优化策略不同：性能优化关注延迟，成本优化关注 token 消耗和模型选择。两者有重叠但目标不同，分开分析更清晰。

## Agent 的成本模型

### 成本构成

一个生产环境的 Agent，成本不是单一的 LLM API 费用。

| 成本项 | 典型占比 | 控制手段 |
|--------|---------|---------|
| LLM 推理 | 70-85% | 模型选择、token 压缩、减少调用次数 |
| 向量检索 | 5-15% | 索引优化、缓存、降维 |
| 外部工具 API | 3-10% | 缓存结果、合并调用 |
| 基础设施 | 2-5% | 实例规格、自动扩缩容 |

LLM 推理是绝对大头，所以成本优化的重心也在 LLM 上。

### 单次请求的成本公式

```
单次请求成本 =
  Σ(每次 LLM 调用 (input_tokens × 输入单价 + output_tokens × 输出单价))
  + Σ(每次检索 (向量数据库单价 × 检索量))
  + Σ(每次工具调用 (外部 API 单价))
```

从公式可以看出，控制成本的核心杠杆有三个：**减少 LLM 调用次数、减少每次调用的 token 量、选更便宜的模型**。

## 成本归因

没有成本归因，你就不知道钱花在哪，也不知道优化后省了多少。

### 归因维度

```
按模型:     gpt-4o: ¥45,000   gpt-4o-mini: ¥8,000
按功能:     订单查询: ¥18,000   客户咨询: ¥22,000
按用户:     企业用户: ¥35,000   个人用户: ¥18,000
按 Agent 阶段: 意图识别: ¥5,000   回答生成: ¥30,000
```

每个维度给出绝对值 + 占比 + 环比变化。环比变化比绝对值更重要——"gpt-4o 成本翻倍"是一个可操作的信号，"gpt-4o 花了 45,000"不是。

### 从 Trace 计算成本

从 Trace 数据中计算成本，需要两个前提：
1. 每条 LLM 调用的 Span 记录了 `input_tokens` 和 `output_tokens`
2. 维护一个模型单价表

```python
# 模型单价表（从 Trace 查询时动态获取也可以）
MODEL_PRICES = {
    "gpt-4o": {
        "input": 2.50 / 1_000_000,
        "output": 10.00 / 1_000_000
    },
    "gpt-4o-mini": {
        "input": 0.15 / 1_000_000,
        "output": 0.60 / 1_000_000
    }
}

def calculate_request_cost(trace: dict) -> dict:
    total_cost = 0.0
    breakdown = {"by_model": {}, "by_step": []}
    for span in trace.get("spans", []):
        if span.get("type") != "llm_call":
            continue
        model = span.get("llm.model", "gpt-4o")
        prices = MODEL_PRICES.get(model, MODEL_PRICES["gpt-4o"])
        in_tok = span.get("llm.input_tokens", 0)
        out_tok = span.get("llm.output_tokens", 0)
        cost = in_tok * prices["input"] + out_tok * prices["output"]
        total_cost += cost
        breakdown["by_model"][model] = \
            breakdown["by_model"].get(model, 0) + cost
    return {"total_cost": total_cost, "breakdown": breakdown}
```

这个函数可以做成一个 UDF，在查询 Trace 时自动算出每条 Trace 的成本。LangFuse 和 LangSmith 都支持这种方式。

### 成本归因的三级粒度

```
请求级 (每 Trace):    单次请求精确成本
  用途: 定位"哪些请求特别贵"、分析成本异常

汇总级 (按维度分组):  按模型/功能/用户聚合
  用途: 成本走读、优化优先级排序

趋势级 (时间序列):    每日/每周成本趋势
  用途: 监控成本变化、发现异常增长
```

建议用 LangFuse 的 dataset + dashboard 组合——Trace 数据自动落入成本表，Grafana 直接查询展示。

## 预算管控

知道了成本归因后，就可以设置预算管控。

### 三层预算模型

```
月度总预算: ¥100,000
├── 模型 A (gpt-4o):         ¥55,000 ← 硬上限
├── 模型 B (gpt-4o-mini):    ¥15,000
├── 向量检索:                 ¥8,000
├── 外部工具:                 ¥5,000
└── 预留弹性:                ¥17,000 (应对突发流量)
```

每项预算设置一个上限，达到上限时触发保护机制。

### 保护机制

```
软限制 (达到预算 80%):
  发送告警通知团队
  新请求自动优先使用更经济的模型
  暂停非核心功能的模型路由

硬限制 (达到预算 100%):
  非核心功能自动降级
  核心功能切换到备用模型（更便宜）
  新请求受速率限制

每请求限制:
  单请求最多调用 10 次 LLM
  单请求最多消耗 50,000 token
  达到限制后自动终止循环并返回友好错误
```

每请求限制是最实用的防线。一个陷入死循环的 Agent 可能在几秒内消耗数百元。有了每请求上限，损失被限制在一个请求的代价内。

### 优化效果的量化

每次优化后，用 Trace 数据量化效果。不做量化的优化是"自我感觉良好"。

```
优化前: 每请求成本 ¥0.32, 月成本 ¥120,000
优化后: 每请求成本 ¥0.12, 月成本 ¥43,000 (-64%)
```

对比时注意排除流量变化的影响——如果请求量涨了 20%，成本涨 10% 反而是省了。应该对比"每请求成本"而非总成本。

## 优化策略

### 模型路由

性价比最高的优化手段。将简单请求分配给更便宜的模型：

```
用户输入 → 分类器
  ├── 简单任务 → gpt-4o-mini (成本降低 80-90%)
  └── 复杂任务 → gpt-4o
```

实践中 60-70% 的请求可以用小型模型处理。分类器本身可以是规则或一个极小的分类模型。

```python
def route_to_model(user_input: str, trace_context: dict) -> str:
    # 规则分类
    if len(user_input) < 100 and "查询" in user_input:
        return "gpt-4o-mini"
    if "分析" in user_input or "对比" in user_input:
        return "gpt-4o"
    # Trace 上下文辅助判断
    if trace_context.get("user_tier") == "premium":
        return "gpt-4o"  # 高价值用户给更好的模型
    return "gpt-4o-mini"
```

路由决策本身要低开销——不能为了省钱而引入额外 LLM 调用。用规则或简单分类器，不要用 LLM 去做路由分类。

### Prompt 压缩

| 手段 | 成本节省 | 质量风险 |
|------|---------|---------|
| 精简 system prompt（删冗余示例） | 5-10% | 极低 |
| 对话历史压缩（限 N 轮+摘要） | 15-30% | 低（摘要信息损失可控） |
| 检索内容裁剪（只保留相关段落） | 10-20% | 低（去掉噪音反而提升质量） |
| 结构化输出（JSON 代替自然语言） | 5-10% | 低 |

对话历史压缩是效果最好的——它同时减少输入和输出 token（上下文短了，LLM 输出也更精准）。实现方式：

```
Agent 对话到第 7 轮时:
  将第 1-5 轮 → 用 LLM 或规则生成摘要
  保留第 6-7 轮 → 原始内容
  新的 prompt = 摘要 + 最近 2 轮
```

### 缓存

| 缓存层次 | 缓存对象 | 命中率 | 节省效果 |
|---------|---------|-------|---------|
| Prompt KV Cache | system prompt 的 KV | 50-70% | 减少 TTFT，不减少 token 计费 |
| 检索结果缓存 | query → 检索结果 | 40-60% | 完全避免检索费用 |
| 工具结果缓存 | 相同 API 请求 → 结果 | 20-40% | 避免外部调用费用 |

提示：不要缓存 LLM 响应。同样的问题在不同上下文中应该给出不同回答。Agent 场景下响应缓存的语义风险大于成本收益。

### 减少 LLM 调用次数

**规则前置**：能用 if 判断的，不交给 LLM。一个 if 语句成本为零。

```
用户说"退款" → 规则匹配"退款"关键词 → 直接路由到退款 Agent
用户说"你好" → 规则匹配问候 → 直接返回"您好，请问有什么可以帮助您的？"
```

这两种场景在客服 Agent 中占比可能达到 20-30%。它们不需要 LLM，一个简单的模式匹配就能处理。

**合并多次调用为一次**：有些 Agent 将一次任务拆成了多次 LLM 调用——先识别意图、再提取参数、再生成回答。如果模型能力够，可以合并为一次调用，让 LLM 一次性输出结构化结果。

**有限状态机替代 LLM 循环**：对于固定流程（如"下单→支付→确认"），用状态机处理，LLM 只负责非结构化的输入理解，流程控制交给代码。

### 各策略的投入产出对比

| 策略 | 实现成本 | 成本节省 | 推荐顺序 |
|------|---------|---------|---------|
| 模型路由 | 低（规则分类） | 40-60% | 1 |
| Prompt 压缩 | 低（改 prompt） | 15-30% | 2 |
| 每请求限制 | 低（加几行代码） | 防止极端损失 | 3 |
| 规则前置 | 中（识别可规则化的场景） | 10-30% | 4 |
| 检索/工具缓存 | 中（加缓存层） | 10-20% | 5 |
| 合并 LLM 调用 | 中（改 Agent 逻辑） | 10-30% | 6 |
| 有限状态机 | 高（重构） | 30-80% | 7 |

模型路由 + Prompt 压缩 两项低成本的优化能覆盖 50-70% 的成本节省空间。建议从这两项开始。

## 案例：一次完整的成本优化

### 背景

客服 Agent，日活 10 万用户，日均 30 万请求。

### 现状测量

从 Trace 数据提取基线：

```
月成本: ¥120,000
每请求平均成本: ¥0.32
每请求平均 token: 4,500
每请求平均 LLM 调用次数: 3.5

成本构成:
  gpt-4o: ¥108,000 (90%)  ← 大头
  gpt-4o-mini: ¥12,000 (10%)
```

### 成本归因分析

按模型维度聚合：

```
gpt-4o: ¥108,000 / 月 (90%)
  └── 简单查询 (60% 的请求): ¥64,800  ← 这部分不应该用 gpt-4o
  └── 复杂分析 (40% 的请求): ¥43,200

gpt-4o-mini: ¥12,000 / 月 (10%)
```

简单查询占了 gpt-4o 成本的 60%——改进空间明确。

按功能维度聚合：

```
订单查询: ¥36,000  ← 这个功能请求最多、场景最固定
客户咨询: ¥48,000
退款处理: ¥24,000
其他:     ¥12,000
```

### 优化方案

```yaml
优化 1: 模型路由 (成本节省: ±50%)
  实施: 简单查询(余额、订单状态) → gpt-4o-mini
  预计: 60% 的请求从 gpt-4o 切换到 gpt-4o-mini

优化 2: 对话历史压缩 (成本节省: ±15%)
  实施: 限制 6 轮，超出部分摘要
  预计: 每请求 token 从 4,500 → 2,800

优化 3: 每请求限制 (成本节省: 防止极端损失)
  实施: 单请求最多 10 次 LLM 调用，50,000 token 上限
```

### 效果量化

```
优化前:   每请求 ¥0.32, 月成本 ¥120,000
优化后:   每请求 ¥0.12, 月成本 ¥43,000
节省:     64%

具体变化:
  gpt-4o 调用占比: 90% → 35%
  每请求 token: 4,500 → 2,800
  每请求 LLM 调用次数: 3.5 → 2.0
```

## 总结

成本优化的核心也是三步：**归因（从 Trace 计算成本分布）→ 识别（找到占比最大的项）→ 优化（从投入产出最高的项开始）**。

成本管理的三条防线：预算分配（月度总额按模型/功能拆分）→ 保护机制（软硬限制 + 每请求上限）→ 持续优化（定期回归，确保优化效果不退化）。

**不要为了省钱牺牲核心体验。** 目标是在成本和质量之间找到最佳平衡点。先让成本可见，再决定优化策略。
