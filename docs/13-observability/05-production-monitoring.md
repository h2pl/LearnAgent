## Agent 的 SLI 和 SLO

SRE 的核心方法是定义 SLI（服务水平指标）→ 设定 SLO（服务水平目标）→ 基于 SLO 设计告警。

### Agent 系统的 SLI

传统 SLI 是延迟、错误率、吞吐量。Agent 系统需要额外补充"任务语义层"的 SLI：

```
系统层 SLI:
  - 请求成功率: HTTP 200 / 总请求数
  - P95 响应时间
  - 服务可用性: (1 - 不可用分钟 / 总分钟)

Agent 专用 SLI:
  - 任务完成率 (TSR): 成功完成的任务 / 总任务数
  - 工具调用准确率: 调用正确的工具 / 总工具调用
  - 人工介入率: 需要转人工的会话 / 总会话
  - 用户满意度: 好评 / (好评 + 差评)
  - 成本效率: 每成功任务的 token 消耗

质量层 SLI:
  - 输出质量评分: LLM-as-Judge 在抽样上的平均值
  - 回答一致性: 相同输入下输出的语义相似度
  - 幻觉率: 抽样中检测到幻觉的比例
```

这里的关键是区分"系统健康"和"任务健康"。HTTP 200 不代表任务完成了——Agent 可能返回了一个"对不起我做不到"但状态码是 200。**Task-level SLI 比 System-level SLI 更接近用户体验。**

### SLO 设定

```
严格模式 (核心业务，如金融客服):
  P95 延迟 < 3s (99% 的时间)
  TSR > 95%
  人工介入率 < 5%

宽松模式 (辅助工具，如内部知识库):
  P95 延迟 < 8s
  TSR > 80%
  人工介入率 < 20%
```

SLO 不是拍脑袋决定的。从 Trace 数据中计算当前的"可达水平"，然后设定 SLO = 当前水平 + 合理的改进目标。

### 错误预算

错误预算 = 1 - SLO。如果 SLO 是 99%，错误预算就是 1%。一个月 43,200 分钟，你有 432 分钟的"犯错额度"。

```
每月错误预算: 432 分钟
已消耗: 120 分钟 (当前用了 28%)
剩余: 312 分钟

消耗速度: 5 分钟/天 (按当前趋势，60 天后耗尽)
```

错误预算耗尽的速度决定了告警的紧急程度。如果消耗速度远高于预期，即使绝对值还小，也需要关注。

## 仪表盘设计

监控仪表盘不是把所有指标堆在一起。好的仪表盘分三层。

### 第一层：全局健康墙 (Big Screen)

每天早上第一眼看的，不超过 6 个数字：

```
┌─────────────────────────────────────┐
│  请求量   P95延迟   TSR    成本/天   │
│  32,000   2.1s     94.2%  ¥4,200   │
│  ↑ 12%    ↑ 8%     ↓ 0.3%  ↑ 5%   │  ← 趋势箭头
├─────────────────────────────────────┤
│  人工介入率  Top 错误                │
│  3.1%      tool_timeout (38%)       │
│  → 0%                              │
└─────────────────────────────────────┘
```

每个数字都有"当前值"+"环比变化"。趋势比绝对值重要——TSR 从 94% 降到 93% 本身不可怕，但如果连续 3 天都在降，就值得关注。

### 第二层：维度拆解

当发现某个指标异常时，点进去看维度拆解：

```
TSR: 94.2%
├── 按功能: 订单查询 98% | 取消订单 92% | 退款 87% ← 退款是短板
├── 按模型: gpt-4o 95% | gpt-4o-mini 92%
└── 按时段: 9-12点 96% | 14-18点 93% | 20-23点 88% ← 晚上效果差
```

维度拆解的目标是**把总量问题变成具体问题**。"TSR 降了"不是一个可操作的信息，"退款功能的 TSR 降了 5%"才是一个可操作的信息。

### 第三层：Trace 搜索

当维度拆解定位到具体问题后，进入 Trace 搜索。按条件过滤并查看单个请求的完整执行过程：

```
条件: 功能=退款 AND 时间=过去1小时 AND TSR=失败
结果: 最近 50 条失败 Trace
```

点开一条 Trace 看完整过程——"原来 Agent 在退款前没有检查订单状态，对已发货的订单发起了退款，工具返回了一个不友好的错误，Agent 没有处理这个错误就直接告诉用户失败了"。

从仪表盘到 Trace 的**下钻路径**是监控体系的核心体验——任何一个异常数字，最多点击 3 次就能看到具体的失败 Trace。

### 推荐的面板组合

一个 Grafana 仪表盘至少包含：

| 面板 | 类型 | 分组 | 说明 |
|------|------|------|------|
| 请求量 | 折线图 | API/功能 | 整体流量趋势 |
| P50/P95/P99 延迟 | 折线图 | 功能 | 注意 P95 和 P99 的差距 |
| TSR | 折线图 | 功能+模型 | 最重要的业务指标 |
| 错误率 | 堆叠面积 | 错误类型 | 区分系统错误和任务错误 |
| Token 消耗 | 折线图 | 模型 | 每日 + 累计 |
| 成本 | 柱状图 | 模型/功能 | 按维度聚合 |
| 用户满意度 | 折线图 | 无 | 每日评分趋势 |
| 人工介入率 | 折线图 | 功能 | 异常上升代表 Agent 处理能力下降 |
| 最慢操作 Top 10 | 列表 | Span 类型 | 实时显示最慢的操作 |

## 告警规则设计

### 什么需要告警

告警设计的第一原则：**只告警需要人处理的事情。** 不需要人处理的事情，不要发告警。

```
需要告警: TSR 从 94% 降到了 85%
不需要告警: CPU 超过 70% (让它自动扩容就行)
```

Agent 系统特有的告警场景：

**场景一：TSR 突降**

```
条件: TSR 的 5 分钟移动平均 < 阈值
阈值: 较过去 1 小时下降 > 5%
响应: 检查最近的部署、模型变更、工具变更
```

**场景二：死循环检测**

```
条件: 单请求内同一工具调用次数 > 阈值 (建议 5 次)
阈值: 连续 3 个请求触发
响应: 检查 Agent 的终止条件
```

**场景三：成本异常**

```
条件: 每请求 token 消耗较 7 日均值 > 2 倍
阈值: 持续 10 分钟
响应: 检查上下文管理、模型路由
```

**场景四：输入漂移**

```
条件: 用户输入的主题分布发生变化
阈值: 某类主题占比翻倍
响应: 检查是否需要补充该类用例的训练/提示数据
```

**场景五：模型响应模式变化**

```
条件: LLM 输出长度、格式、拒绝率突然变化
阈值: 各项指标较 24h 前变化 > 30%
响应: 检查模型版本是否被更新
```

### 告警分级

| 级别 | 含义 | 响应时间 | 通知方式 |
|------|------|---------|---------|
| P0 | 核心功能不可用、TSR 跌破底线 | 立即 | 电话 + IM |
| P1 | 核心功能受损、TSR 显著下降 | 15 分钟 | IM |
| P2 | 非核心功能异常、指标预警 | 2 小时 | 邮件 |
| P3 | 信息通知、无影响 | 24 小时 | 邮件 |

不要给太多 P0。每个 P0 都会制造人的警觉疲劳。P0 应该是在凌晨 3 点也会把你叫醒的级别——如果不够这个标准，就不要设 P0。

## 告警疲劳的防御

告警疲劳是监控系统最大的敌人——当告警太多，人会选择性忽略。三个防御手段：

### 手段一：聚合相似告警

同一个根因不要重复告警。如果某个工具 API 挂了三分钟，每分钟收到一条"TSR 下降"的告警，这就是噪音。

```
错误做法:
  09:01 TSR 下降告警
  09:02 TSR 下降告警
  09:03 TSR 下降告警 (同一根因，三条告警)

正确做法:
  09:01 TSR 下降告警 (触发)
  09:05 TSR 下降告警 (5 分钟内未恢复，升级)
```

### 手段二：依赖告警

当一个告警的根因可能是另一个告警时，只发根因的告警。

```
例子:
  Agent 依赖的 LLM API 挂了 → 所有请求都失败
  此时发了 5 条告警:
    - "LLM API 不可用" ← 这是根因
    - "TSR 下降"
    - "错误率上升"
    - "延迟上升"
    - "成本为零"(因为没有请求成功)

  应该只发一条: "LLM API 不可用"
```

实现方式：告警规则中设置依赖关系。当根因告警触发时，抑制所有下游告警。

### 手段三：冷静期

同一个告警规则触发后，设定一个"冷静期"（通常是 30 分钟）。冷静期内即使条件仍然满足，也不再重复告警。

```
告警: "死循环检测"
触发: 09:00
冷静期: 30 分钟
下次触发: 09:30 (如果仍然异常)
```

## Runbook：每个告警的配套文档

每个告警规则必须有对应的 Runbook（运维手册）。没有 Runbook 的告警等于没说——"TSR 降了，然后呢？"

### Runbook 模板

```markdown
# Runbook: TSR 下降

## 检查步骤
1. 登录 Grafana → 查看 TSR 面板 → 按功能/模型/时段拆解
2. 查看最近 1 小时的 Trace 搜索 → 过滤失败的请求
3. 查看最近部署记录 → 是否有变更？
4. 查看外部依赖状态 → LLM API / 工具 API 是否正常？

## 常见原因


| 现象 | 原因 | 处理 |
|------|------|------|
| 所有功能 TSR 都降 | LLM API 异常 | 检查 LLM 状态页 |
| 特定功能 TSR 降 | 该功能的工具或 prompt 有变更 | 回滚最近变更 |
| 特定模型 TSR 降 | 模型版本更新 | 切换到旧版本或联系供应商 |
| 特定时段 TSR 降 | 用户输入模式变化 | 补充该时段的评测集 |

## 恢复步骤
1. 如果是部署导致：回滚到上一个版本
2. 如果是外部依赖：联系供应商，切换备用方案
3. 如果是 prompt 问题：回滚 prompt 变更

## 升级流程
- 15 分钟未恢复 → 通知 Tech Lead
- 30 分钟未恢复 → 全组响应
- 60 分钟未恢复 → 考虑降级服务
```

### Runbook 的维护

Runbook 不是写一次就完的。每次事故处理后更新 Runbook：

```
事故: 退款 TSR 突降，根因是新版 OpenAI 模型不理解中文退款政策
更新 Runbook: 增加一条 "模型更新后检查中文场景的 TSR"
```

## 案例：一套完整的 Agent 监控体系

### 背景

电商平台的客户服务 Agent，日均处理 30 万请求。

### 指标定义

```
SLI:
  - TSR ≥ 95%
  - P95 延迟 ≤ 4s
  - 人工介入率 ≤ 5%
  - 日成本 ≤ ¥5,000

SLO:
  - 月度 TSR ≥ 93%
  - 月度 P99 延迟 ≤ 10s
  - 月度可用性 ≥ 99.9%
```

### 仪表盘

三个 Grafana Dashboard：

**团队大屏 (Team Wall)**：TSR + P95 延迟 + 日成本 + 人工介入率。每天站会前看一眼，持续一周下降趋势代表需要关注。

**运维面板 (Ops Dashboard)**：全量指标 + 维度拆解 + 错误分布。出现告警时第一个打开的面板。

**Trace 浏览器**：按 Trace ID 搜索、按条件过滤。排查问题时使用。

### 告警规则

```
P0:
  - TSR < 85% (持续 3 分钟)
  - 服务不可用 (up == 0)

P1:
  - TSR 较过去 1 小时下降 > 5%
  - P95 延迟 > 8s (持续 5 分钟)
  - 死循环检测 (同一工具调用 > 5 次，连续 3 个请求)

P2:
  - 日成本 > 预算 80%
  - 人工介入率 > 10%
  - 每请求 token 消耗翻倍
```

### Runbook

每个 P0/P1 告警有完整的 Runbook，包括检查步骤、常见原因、处理方式、升级流程。

### 效果

| 指标 | 实施前 | 实施后 | 变化 |
|------|--------|--------|------|
| 故障平均恢复时间 (MTTR) | 45 分钟 | 12 分钟 | -73% |
| 告警数量/天 | 47 条 | 8 条 | -83% |
| 错误预算消耗 | 6.5%/月 | 2.1%/月 | -68% |
| 人工介入率 | 8% | 3.2% | -60% |

好的监控体系不只缩短了恢复时间，还通过 Runbook 沉淀了知识，让团队对系统的理解越来越深。

## 总结

生产监控不是简单挂几个图表。它需要一套完整的体系：

**指标层**（SLI/SLO 定义）→ **可视化层**（三层仪表盘：总览→拆解→Trace）→ **告警层**（分级告警 + 防疲劳机制）→ **响应层**（Runbook 沉淀知识）。

Agent 系统独有的监控挑战在于：指标需要度量"任务健康"而不仅是"系统健康"；告警需要检测非确定性行为（死循环、模型漂移）而不只是 HTTP 错误；Runbook 需要覆盖 LLM、工具、检索等多元故障模式。
