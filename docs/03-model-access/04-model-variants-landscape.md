## 变体全景：6 个维度

"变体"（Variants）在 LLM 生态里指**针对特定目标做了专门优化的模型版本**。从大类上分：

| # | 维度 | 解决的问题 | 典型变体 | 如何得到 |
|---|------|----------|---------|---------|
| 1 | **训练阶段** | 让模型从"续写句子"变成"听话做事" | Base / Instruct / Chat（开源仍用三分法）| **后训练**：在 Base 模型上做 SFT → RLHF/DPO |
| 2 | **能力专精** | 某个能力做到极致 | Code / Math / Vision / Function Calling 专用 | **后训练**：用领域数据（代码/数学/图文）做专项 SFT + 对齐 |
| 3 | **部署形态** | 适配不同硬件和资源 | 量化、合并、剪枝 | **后处理**：对已有权重做数学变换（量化/合并/剪枝），不训练 |
| 4 | **蒸馏** | 用小模型"继承"大模型能力 | DeepSeek-Distill / Phi / SmolLM 系列 | **教师监督训练**：学生模型从头训练，用教师输出的软标签做监督——既不是纯后训练，也不是纯后处理 |
| 5 | **专用模型家族** | 不同任务用不同范式 | Embedding / Rerank / STT / TTS | **独立训练**：完全不同的架构（Encoder/Cross-Encoder/声学模型），从头训练 |
| 6 | **行业 / 场景** | 领域知识 + 合规 | 金融 / 医疗 / 法律行业模型 | **后训练**：通用模型 + 行业数据 SFT，注入领域知识 |

**与 03 章的关系**：03 讲的是"同家族内的尺寸 + 上下文窗口"——那不算"变体"，是"同一变体内的不同配置"。本篇讲的 6 个维度是"真正意义上不同的变体"。

> **一句话区分**：变体不是营销，是 LLM 生态分工协作的必然产物——通用大模型负责"什么都能做"，变体负责"把某件事做到极致"。

## 变体从哪来：三种来源

很多读者一上来看到 6 个维度、几十种变体，第一反应是："这些不都是微调出来的吗？"**不是**。这 6 个维度的变体，来源完全不同。先搞清楚来源，再看细节，就不会晕。

### 来源一：后训练（Post-Training）—— 从 Base 加工而来

**本质**：同一套架构和参数，通过额外的训练阶段改变行为。

| 典型变体 | 训练方式 | 一句话说明 |
|---------|---------|-----------|
| Instruct / Chat / Reasoning | SFT → RLHF/DPO | 教模型"怎么回答"和"什么不该回答" |
| Code / Math / Vision 专用 | 在通用模型基础上做领域 SFT + 对齐 | 用代码/数学/多模态数据替换通用对话数据，重新训练后几层 |
| 金融 / 医疗 / 法律模型 | 通用模型 + 行业数据 SFT | 注入行业知识，同时保持通用能力 |

**关键特征**：变体和 Base 模型是**同一架构、同一参数空间**——你可以把 Instruct 理解为"Base 的参数经过 SFT 后的新位置"。这就是为什么 HuggingFace 上 Llama-3-8B 和 Llama-3-8B-Instruct 的架构文件（`config.json`）完全一样。

### 来源二：后处理（Post-Processing）—— 数学变换，不训练

**本质**：对已有模型的权重做数学操作，**不接触任何数据、不跑梯度下降**。

| 典型变体 | 技术手段 | 一句话说明 |
|---------|---------|-----------|
| 量化（Q4_K_M） | 把 FP16 权重映射到 4-bit 整数 | 像把 WAV 音乐压缩成 MP3——内容一样，精度降低 |
| 合并（Merge） | 两个模型权重做线性插值 | 像把两杯不同口味的咖啡倒在一起 |
| 剪枝（Pruning） | 把接近 0 的权重直接删除 | 像修掉盆景里多余的枝叶 |

**关键特征**：后处理变体**不改变模型知道什么**，只改变"怎么存"和"怎么算"。量化后的模型还是同一个模型，只是数字精度低了。这和后训练有本质区别——后训练让模型"学会新行为"，后处理只是"换个包装"。

### 来源三：独立训练（Independent Training）—— 从头另造

**本质**：不是从 LLM 改出来的，而是**完全不同的架构、从头训练**的模型。

| 典型变体 | 架构差异 | 一句话说明 |
|---------|---------|-----------|
| Embedding（BGE-M3） | 双塔 Encoder，不是 Decoder | 把文本变成向量——和生成文本是完全不同的数学任务 |
| Rerank（BGE-reranker） | 交叉编码器（Cross-Encoder） | 判断"问题 A 和文档 B 有多相关"——不是生成 |
| STT（Whisper） | 编解码器（Encoder-Decoder） | 语音→文字，输入是音频频谱图 |
| TTS（CosyVoice） | 声学模型 + 声码器 | 文字→语音，输出是波形或 mel 谱 |

**关键特征**：这些模型和 LLM 的关系，就像冰箱和汽车的关系——都叫"家电"和"交通工具"，但内部结构完全不同。你**不能**把 Llama 3 "微调"成 Embedding 模型，就像不能把汽车改装成冰箱。

### 一个容易混淆的特例：蒸馏

蒸馏（Distillation）是独立的维度4，但它的来源有点特殊——**既不是纯后训练，也不是纯后处理**：

- 学生模型是**从头训练**的（不是微调教师模型）
- 但损失函数用教师的输出当"软标签"来监督
- 所以它是**独立训练 + 教师监督**的混合体

你可以理解为：学生不是继承教师的参数，而是"旁听教师的课"来自己学。

### 三种来源总览

```
后训练（Post-Training）     后处理（Post-Processing）     蒸馏（Distillation）       独立训练（Independent）
       ↓                            ↓                          ↓                          ↓
  同架构+新数据训练          权重数学变换，不训练         教师监督+学生从头训练        不同架构从头训练
  ──────────────────────────────────────────────────────────────────────────────────────────────────
  Instruct/Chat           量化(Q4_K_M)/合并/剪枝       DeepSeek-Distill            Embedding/Rerank
  Code专用/Math专用           不改变模型能力              Phi/SmolLM系列              STT/TTS
  行业模型                   只改变存储和计算方式          小而精，继承大模型能力        和LLM完全不同的架构
```

**记住这个顺序**：先问"这个变体是从哪来的"——是后训练、后处理、还是独立训练？——再看具体细节。下文 6 个维度会按这个框架展开。

## 变体维度 1：训练阶段

同一架构在训练的不同阶段会产出不同变体。这是 HuggingFace 上下载开源模型时**最容易踩坑**的地方。

### 三种基础变体

| 变体 | 训练阶段 | 能力特点 | 能否直接用于 Agent |
|------|---------|---------|------------------|
| **Base 模型** | 仅预训练 | 有知识，但不听指令。给它"写一首诗"，它会续写成"写一首散文、写一部小说..." | ❌ 不能 |
| **Instruct 模型** | 预训练 + SFT | 听指令，能问答，但可能不够安全或礼貌 | ✅ 基础可用 |
| **Chat 模型** | 预训练 + SFT + RLHF/DPO | 安全、有用、诚实，经过人类偏好对齐 | ✅ **生产首选** |

### 这些变体是怎么从训练中产生的

三种变体不是"同一个模型用不同数据微调了一下"那么简单——**每一阶段都改变了模型的目标函数，从根本上重塑了行为模式**。

**Base 模型为什么只会续写？**

预训练的目标是**自回归预测下一个词**（Autoregressive Next-Token Prediction）。模型在万亿 Token 上优化，参数编码的是"给定前文，下一个词最可能是什么"的概率分布。当你输入"请写一首关于春天的诗"，模型在训练数据中见过无数类似的排比结构——"请写一首关于夏天的诗、请写一首关于秋天的诗..."——概率分布最匹配的模式是**继续列举**而非**执行指令**。这不是模型"没理解"，而是目标函数的必然产物：它从未被要求"回答"，只被要求"续写"。

**Instruct 模型为什么能听话？**

SFT（Supervised Fine-Tuning）改变了目标函数：从"预测下一个词"变成**最大化指令-回答对的条件似然**（Maximize Conditional Likelihood of Instruction-Response Pairs）。训练数据是人工编写的 `(指令, 回答)` 对，模型在优化过程中，参数逐渐编码了新的概率分布：看到指令格式的输入时，生成回答格式的输出。这相当于在参数层面建立了一条**从"指令语义"到"回答行为"的映射通路**。但 SFT 只教了"怎么回答"，没有教"什么不该回答"——所以 Instruct 模型可能直接回答危险问题。

**Chat 模型为什么更安全？**

RLHF/DPO 再次改变了目标函数。不再优化"回答像不像示范数据"，而是优化**人类偏好的概率**（或奖励模型的分数）。训练数据是人类对同一问题的多个回答做排序（"回答 A 比回答 B 更好"）。模型在优化过程中学会了：直接回答"如何制造炸弹"会得低分（人类会把它排在最后），而拒绝并解释原因会得高分。在参数层面，这表现为**某些生成路径的概率被系统性地压低**（如直接给出危险信息），另一些路径的概率被抬高（如拒绝+解释）。这不是"增加了安全知识"，而是**行为策略的重塑**——同样的知识，模型选择不同的表达方式。

**推理模型（Reasoning）的历史特殊路径**

2024 年的 o1 和 R1 作为独立变体出现，它们的训练不是标准三阶段的简单延伸，而是**用特殊数据重构了 SFT 和对齐阶段**：

- **SFT 阶段**：训练数据不是普通的 `(问题, 回答)` 对，而是 `(问题, 详细思维链, 回答)` 三元组。模型被要求先输出内部思考过程（"让我先分析...""我需要考虑..."），再给出最终答案。
- **对齐阶段**：奖励模型不是评估"回答是否礼貌/有用"，而是评估**最终答案的正确性**。强化学习优化的目标是"思考过程是否导向了正确答案"，而非"思考过程本身是否优美"。

到 2026 年，这种能力已统一进主流模型（通过 `reasoning_effort` 参数控制），但训练原理仍然适用：推理能力是从**特殊的思维链 SFT 数据**和**以正确性为导向的对齐目标**中涌现的。详细的调用方式、成本权衡和 Agent 规划层实践见 [深度思考与推理能力](./05-deep-thinking-and-reasoning.md)。

### 实际影响

以 Llama 3 为例，HuggingFace 上你会看到：

- `Meta-Llama-3-8B` → **Base 模型**（研究用）
- `Meta-Llama-3-8B-Instruct` → **Instruct 模型**（Agent 开发用这个）
- `Meta-Llama-3-8B-Instruct-GGUF` → 量化版本（本地部署用）
- `Meta-Llama-3-8B-Instruct-Q4_K_M` → 4-bit 量化（消费级 GPU）

> **Agent 开发永远下载 Instruct 或 Chat 版本**。Base 模型是给研究人员做进一步微调的半成品——它不会回答你的问题，只会续写你的句子。

## 变体维度 2：能力专精

### Code 专用模型

通用模型能写代码，但 **Code 专用模型在代码任务上更快、更准、更便宜**。

| 模型 | 来源 | 规模 | 特点 | Agent 用途 |
|------|------|------|------|----------|
| **Codestral 2** | Mistral | 7B / 22B | 开源，工具调用友好 | IDE 补全、代码 Agent |
| **DeepSeek-Coder-V3** | DeepSeek | 16B / 236B MoE | 中英双语代码，SWE-bench 强 | 复杂重构、跨文件理解 |
| **Qwen3.6-Coder** | 阿里 | 0.5B-32B | 尺寸齐全，可本地部署 | 多场景覆盖 |
| **Code Llama 4** | Meta | 7B / 13B / 34B / 70B | 经典选择 | 已逐渐被新一代取代 |
| **GPT-5.5 / Claude Opus 4.7** | 商用 API | — | 综合能力含代码 | 中等代码任务 |

**什么时候用 Code 专用模型？**

- IDE 插件 / 代码补全（要求 < 100ms 响应，专用小模型更快）
- 训练数据高度专业（编译原理、底层优化）
- 离线环境无法调 API

**什么时候通用模型就够了？**

- Agent 工具调用（GPT-4o/Claude 的工具调用已经很成熟）
- 简单的代码解释、转换、生成
- 提示词工程层面已经能解决

### Math 专用模型

| 模型 | 规模 | 特点 | 适用 |
|------|------|------|------|
| **Qwen3.6-Math** | 1.5B / 7B / 72B | 中文数学题解强 | 教学、题库 |
| **NuminaMath-2** | 7B | 竞赛数学 | 数学竞赛 |
| **DeepSeek-Math-V2** | 7B | 学术数学 | 论文级数学 |

**重要提醒**：GPT-5.5 / Claude Opus 4.7 / DeepSeek V4-Pro 等闭源旗舰**默认就是 Math/Logic 推理能力**——通过 `reasoning_effort=high` 或 `effort=max` 启用深度思考后，能力远超专门 Math 模型。Math 专用模型适合**离线 + 推理成本极度敏感**的场景，否则直接用闭源旗舰 + 深度思考。

### 多模态变体（Vision）

能"看懂"图片的模型——同一架构基础上加了视觉编码器。

| 模型 | 视觉能力 | 上下文 | 适合场景 |
|------|---------|-------|---------|
| **GPT-5.5**（Vision） | ★★★★★ | 1M | 通用图文理解 + 思考 |
| **Claude Opus 4.7 / Sonnet 4.6** | ★★★★★ | 1M | 文档+图片分析 + adaptive thinking |
| **Gemini 3.1 Pro** | ★★★★★ | 1M+ | 长视频、超长文档 + 推理 |
| **Qwen3.6-VL** | ★★★★ | 262K | 国产开源首选，多模态 |
| **LLaVA-NeXT** | ★★★ | 32K | 本地部署 |

> 注意：**不是所有同族模型都支持 Vision**。例如 Llama 3 纯文本版不支持图片，Llama 3.2 Vision 才支持。选型时要看清楚能力矩阵。

### Function Calling 优化版

工具调用（Function Calling）是 Agent 的核心能力。各家模型在工具调用上的稳定性差异巨大：

| 模型 | 工具调用能力 | JSON 稳定性 |
|------|------------|------------|
| **GPT-5.5 系列** | ★★★★★ | 99%+ |
| **Claude Opus 4.7 / Sonnet 4.6** | ★★★★★ | 99%+ |
| **Gemini 3 Flash / Pro** | ★★★★★ | 99%+ |
| **Qwen3.6 / DeepSeek V4** | ★★★★ | 95%+ |
| **Llama 4 Scout / Maverick** | ★★★ | 90%+ |
| **Mistral Large 3** | ★★★ | 85%+ |

**Agent 框架（LangChain、AutoGen）对工具调用格式要求严格**——某个 JSON 字段写错就解析失败。建议优先选择 ★★★★ 及以上的模型；如果只能选小模型，**用专门微调过的 Function Calling 版本**（如 Hermes 系列、ToolACE）。

## 变体维度 3：部署形态

部署形态变体主要解决**"我的 GPU 跑不动原版模型"**或**"我想要更小的模型"**的问题。

### 量化（Quantization）

将模型权重从高精度（FP16/FP32）压缩到低精度（4-bit/8-bit），**减少显存占用、加快推理速度**。

| 格式 | 精度 | 显存节省 | 质量损失 | 典型工具 |
|------|------|---------|---------|---------|
| FP16（原始） | 16-bit | 1x | 0% | — |
| **BF16** | 16-bit | 1x | < 0.1% | HuggingFace |
| **INT8** | 8-bit | ~2x | < 1% | bitsandbytes |
| **Q8_0 (GGUF)** | 8-bit | ~2x | < 1% | llama.cpp / Ollama |
| **Q4_K_M (GGUF)** | 4-bit 混合 | ~4x | 1-2% | Ollama（推荐） |
| **AWQ** | 4-bit | ~4x | 1-3% | AutoAWQ |
| **GPTQ** | 4-bit | ~4x | 1-3% | GPTQ-for-LLaMa |
| **Q2_K (GGUF)** | 2-bit | ~8x | 5-10% | 极限压缩 |

**选型建议**：

- 显存充足（4x 模型大小）→ **FP16 / BF16**（最准）
- 显存紧张但要保留质量 → **INT8**
- 消费级 GPU 跑 7B-13B 模型 → **Q4_K_M GGUF**（性价比之王）
- 24GB 显存想跑 70B → **AWQ 4-bit** + offload

> **Q4_K_M 是消费级 GPU 跑大模型的"标配"**——4 倍显存节省，质量损失 < 2%，几乎人人都在用。

#### 技术原理：从 FP16 到 4-bit 是怎么做的

量化不是简单"把数字砍半"——它是一组**映射函数**，把高精度浮点数映射到低精度整数，同时尽量减少信息损失。

**线性量化**（最常用）：

```
原始权重 w (FP16) → 缩放 + 零点偏移 → 量化值 q (INT4/INT8)
q = round((w - zero_point) / scale)
w ≈ q * scale + zero_point
```

- **Scale**：量化区间的缩放因子，决定 4-bit 能表示的数值范围
- **Zero-point**：零点偏移，处理权重分布不对称（如大部分权重集中在 -0.1 到 +0.1 之间）
- **非对称量化**：不同层、甚至不同权重矩阵用不同的 scale 和 zero-point，因为各层的权重分布差异巨大

**GGUF 的"混合量化"**（以 Q4_K_M 为例）：

不是把模型的所有权重统一量化到 4-bit。GGUF 把权重分成**256 个一组**（block），每组内部：

1. 计算这 256 个权重的最小值和最大值
2. 用 6-bit 存储这个 block 的 scale 和 min（这是"量化参数的量化"）
3. 用 4-bit 存储每个权重在这个 block 内的相对位置

**为什么叫"混合"**：不同部分用不同精度。attention 的 Q/K/V 矩阵通常用更高精度（6-bit），FFN 的权重用 4-bit——因为 FFN 参数更多但对精度更不敏感。

**质量损失来源**：

- 极端值（outliers）被压缩后失真：少数权重特别大（如某些 attention 的特定 head），4-bit 表示范围有限，这些值被截断
- 累积误差：多层 Transformer 堆叠，每层的量化误差传递到下一层被放大
- 激活值量化（AWQ/GPTQ 的进阶）：不仅量化权重，还量化中间激活值，进一步减少显存但误差更大

**AWQ vs GPTQ vs GGUF**：

- **GGUF**（llama.cpp）：运行时动态反量化，对特定权重保护（如 attention 的 outlier 用 8-bit），适合 CPU 和 Apple Silicon
- **AWQ**：推理时保持 4-bit，只在矩阵乘法时反量化——速度更快，但需要 CUDA kernel 支持
- **GPTQ**：一次性把权重量化到 4-bit，用近似 Hessian 矩阵补偿误差——量化后模型文件更小，但精度损失比 AWQ 稍大

## 变体维度 4：蒸馏

把大模型的能力"传授"给更小的模型，让小模型"继承"大模型的部分能力。

- **DeepSeek-V4-Distill-Qwen / Llama 系列** → DeepSeek 把 R1/V4 的推理能力蒸馏到 7B / 32B 模型
- **Qwen3.x 系列** → 阿里用旗舰蒸馏出 0.5B/1.5B/7B/14B/32B/72B 全家桶
- **Phi-4 / Phi-5 系列** → 微软用旗舰模型蒸馏出 3.8B/14B 小模型，性能惊人
- **SmolLM3** → HuggingFace 用 Llama 4 蒸馏到 1.7B/3B

**优势**：同参数量下能力更强（小而精）
**劣势**：能力上限受限于教师模型

#### 技术原理：教师怎么把能力"教"给学生

蒸馏不是"把大模型的参数复制给小模型"——小模型参数数量远少于大模型，根本装不下。蒸馏的本质是**让小模型学习教师模型的"行为分布"**。

**软标签 vs 硬标签**：

假设一个分类任务，真实标签是"猫"。硬标签就是 one-hot：`[1, 0, 0, 0]`（猫、狗、鸟、车）。教师模型输出的概率分布可能是 `[0.7, 0.2, 0.08, 0.02]`——这个分布蕴含了**暗知识**（dark knowledge）：教师认为"狗"和"猫"比"鸟"和"猫"更相似。学生模型直接学习这个概率分布，比只学习"正确答案=猫"获得更多信息。

**温度参数（Temperature）**：

在蒸馏时，教师模型的 softmax 计算中加入温度参数 T：

```python
import torch.nn.functional as F

# 温度 T > 1 时，概率分布更平滑（差异缩小）
# 温度 T = 1 时，标准 softmax
# 温度 T < 1 时，概率分布更尖锐（赢家通吃）
soft_probs = F.softmax(logits / temperature, dim=-1)
```

- **T=2~4**：教师输出更平滑，学生能看到更多类别间的相对关系，传递更多暗知识
- **T=1**：标准推理温度，蒸馏效果较弱
- **实践中**：先用高温训练（T=4），再用低温微调（T=1），最后和真实标签损失联合优化

**损失函数**：

蒸馏训练同时优化两个目标：

1. **蒸馏损失**（KL 散度）：学生输出 vs 教师软标签的分布差异
2. **真实标签损失**（交叉熵）：学生输出 vs 真实标签的硬标签

```python
# 蒸馏训练的损失 = alpha * 蒸馏损失 + (1-alpha) * 真实标签损失
loss = alpha * distillation_loss(student_logits, teacher_logits, T=4) \
     + (1 - alpha) * ce_loss(student_logits, true_labels)
```

- **alpha 通常取 0.5~0.7**：蒸馏损失权重更高，但真实标签提供"保底"
- **为什么有用**：大模型在推理过程中，参数编码了复杂的决策边界。小模型无法直接获得这些参数，但可以学习大模型面对各种输入时的"行为模式"——相当于小模型获得了大模型在参数空间中探索到的"捷径"

**局限性**：

- 学生模型的架构必须和教师兼容（不能让学生用 CNN 学 Transformer 的行为）
- 蒸馏主要传递"行为模式"而非"知识"——如果学生模型容量太小，它只能学到部分模式，复杂推理能力会丢失
- 开源蒸馏版（如 DeepSeek-V4-Distill-Qwen-7B）通常在数学/代码上表现好，但通用对话和创意写作明显弱于教师模型

### 合并（Merge）

把多个微调模型"线性组合"，获得混合能力。

```python
# 用 mergekit 合并两个 LoRA 模型
import yaml
config = """
models:
  - model: model-A   # 偏聊天
    weight: 0.5
  - model: model-B   # 偏代码
    weight: 0.5
merge_method: linear
"""
with open("merge.yaml", "w") as f:
    f.write(config)
# mergekit-yaml merge.yaml merged-model
```

**典型场景**：

- 模型 A 偏聊天 + 模型 B 偏代码 → 合并后既能聊又能写代码
- 不同 DPO 偏好模型 → 合并出"更平衡"的对齐效果

**优势**：低成本获得定制能力
**劣势**：合并效果难预测，需要实验

#### 技术原理：权重空间中的"插值"

模型合并不是在数据层面混合训练数据，而是在**参数层面**混合模型权重。想象两个模型的每个参数都是高维空间中的一个点，合并就是在这两个点之间画一条线，取线上的某个位置作为新模型的参数。

**为什么能工作？**

深度神经网络的损失 landscape 不是混乱的丘陵，而是**有结构的高维盆地**（loss basin）。两个在同一基础模型上微调的模型（如 Llama-3-8B-Instruct 和 Llama-3-8B-Coder），它们的参数通常位于同一个 basin 的不同位置。在这个 basin 内部做线性插值，新模型的损失通常不会剧烈跳变——这就是为什么合并两个微调模型往往能得到"兼具两者能力"的结果。

**常见合并方法**：

| 方法 | 原理 | 适用场景 |
|------|------|---------|
| **Linear** | 简单加权平均：`w_merged = alpha * w_A + (1-alpha) * w_B` | 两个模型能力互补，无冲突 |
| **SLERP**（Spherical Linear Interpolation） | 在球面上做插值，保持向量长度不变 | 防止插值后向量范数塌陷 |
| **TIES**（Trim, Elect Sign & Merge） | 1. 修剪（只保留最重要的参数差异）<br>2. 选举符号（解决冲突：A 想把某参数增大，B 想减小，选票数多的方向）<br>3. 合并 | 两个模型对同一参数有冲突修改时 |
| **DARE**（Drop And REscale） | 随机丢弃一定比例的参数差异，然后放大保留的部分 | 防止合并后模型过度拟合到任意一方 |

**为什么不是任意两个模型都能合并？**

- 不同基础架构（如 Llama 和 Qwen）的参数空间完全不同，强行插值会产生垃圾
- 同一架构但微调差异过大（如一个只训练了 100 步、一个训练了 10000 步），可能位于不同的 basin，插值路径会经过高损失区域
- 使用 mergekit 等工具时，通常需要选择**基于同一基础模型**的微调版本

**实践建议**：合并前先做"线性插值扫描"——从 alpha=0.0 到 1.0 以 0.1 为步长测试，观察在哪一段性能最优。某些任务可能在 alpha=0.3 时效果最好（模型 A 的能力占主导），而不是简单的 0.5 对半分。

### 剪枝（Pruning）

移除模型中"不重要"的神经元或层，让模型更小。

- **结构化剪枝**：移除整个 attention head 或 FFN 层
- **非结构化剪枝**：移除单个权重（更激进但需要稀疏计算支持）

实操中剪枝**不如量化+蒸馏常用**——剪枝后的模型在通用 GPU 上加速不明显。

#### 技术原理：识别"不重要的"参数

剪枝的核心问题是：**怎么判断一个参数（或一组参数）是"不重要"的？**

**重要性评分方法**：

| 方法 | 原理 | 粒度 |
|------|------|------|
| **Magnitude Pruning** | 参数绝对值小的不重要 | 单个权重 |
| **Gradient-based** | 梯度接近 0 的参数对损失影响小 | 单个权重 / 层 |
| **Hessian 近似**（如 OBD/OBS） | 参数对二阶导数（Hessian）贡献小的可以移除 | 单个权重 |
| **Layer-wise Importance** | 整层对输出的贡献小则移除整层 | 层 / attention head |

**结构化剪枝 vs 非结构化剪枝**：

- **结构化剪枝**：移除整个 attention head、FFN 层或 Transformer Block。优点是剪枝后模型仍是稠密矩阵，不需要特殊硬件支持；缺点是粒度粗，每移除一个 head 可能同时损失多种能力。

```python
# 结构化剪枝：移除贡献最小的 20% attention heads
importance = compute_head_importance(model)  # 基于梯度或注意力模式
heads_to_prune = importance.bottom_k(0.2)     # 取最不重要的 20%
model.prune_heads(heads_to_prune)
```

- **非结构化剪枝**：移除单个权重（如把 30% 的权重置零）。优点是粒度细，可以精确控制稀疏度；缺点是剪枝后产生稀疏矩阵，通用 GPU 的稠密矩阵乘法无法直接加速，需要专门的稀疏计算库（如 cuSPARSE）。实践中，**稀疏度需要达到 70% 以上才能弥补稀疏计算的开销**——而 LLM 通常只能剪到 30-40% 稀疏度就开始严重损失质量。

**为什么剪枝不如量化常用？**

| 技术 | 加速方式 | 通用 GPU 支持 | 质量-速度权衡 |
|------|---------|-------------|-------------|
| 量化 | 降低精度，减少内存带宽 | ✅ 原生支持（INT8/INT4 指令） | 优秀（4-bit 损失 < 2%） |
| 结构化剪枝 | 减少层数/head 数 | ✅ 模型变小自然加速 | 中等（剪 20% 损失 5-10%） |
| 非结构化剪枝 | 稀疏矩阵运算 | ❌ 需要稀疏库，稀疏度低时反而更慢 | 较差（稀疏度 < 70% 不划算） |

**结论**：对 LLM 来说，**量化是性价比最高的压缩手段**（4-bit 几乎无损，原生支持），合并是定制能力的最快方式，而剪枝在 LLM 场景下**实用性最低**——除非你有专门的稀疏硬件（如某些推理芯片）。

## 变体维度 5：专用模型家族

LLM 生态里除了"对话生成"模型，还有几个**完全不同的范式**专门解决特定任务。

### 嵌入模型（Embedding）

**把文本变成向量**——用于相似度计算、检索、聚类。

| 模型 | 维度 | 上下文 | 特点 |
|------|------|--------|------|
| **OpenAI text-embedding-3-large** | 3072 | 8K | 综合最强 |
| **BGE-M3** | 1024 | 8K | **中文开源最强** |
| **BGE-large-zh-v1.5** | 1024 | 512 | 中文经典 |
| **M3E** | 768 | 8K | 中文轻量 |
| **Cohere embed-v3** | 1024 | 8K | 多语言商用 |
| **Nomic Embed Text v2** | 768 | 8K | 长文本开源 |

**重要**：**对话模型 ≠ 嵌入模型**！两个完全不同的范式：

| 维度 | 对话模型 | 嵌入模型 |
|------|---------|---------|
| 任务 | 文本生成 | 文本→向量 |
| 结构 | 自回归（Decoder） | 双塔结构（Encoder） |
| 输出 | Token 序列 | 一个定长向量 |
| 用途 | 聊天、推理 | 检索、聚类、推荐 |

> 混用会导致检索效果非常差。例如把 Llama 3 的最后一层 hidden state 当 embedding 用，效果远不如专门的嵌入模型。

### 重排序模型（Rerank）

**对召回结果重新打分排序**——RAG 系统的精排阶段必备。

| 模型 | 规模 | 速度 | 质量 |
|------|------|------|------|
| **BGE-reranker-v2-m3** | 568M | 中 | ★★★★★ 多语言 |
| **BGE-reranker-large** | 560M | 中 | ★★★★ |
| **Cohere Rerank 3** | API | 快 | ★★★★★ |
| **Jina Rerank** | API | 快 | ★★★★ |
| **mxbai-rerank-large-v2** | 1.5B | 中 | ★★★★ |

**RAG 标准两阶段**：

```
用户问题 → 嵌入模型召回 Top 100 → Rerank 精排 Top 5 → LLM 生成答案
            ↑                            ↑
        速度优先                     精度优先
```

Rerank 模型比直接用 LLM 精排**快 10-100 倍**，且更准。

### 语音模型

| 用途 | 模型 | 特点 |
|------|------|------|
| **STT**（语音→文字） | Whisper / Whisper-Large-V3 | OpenAI 开源，多语言 |
| **STT** | Paraformer | 阿里开源，中文优秀 |
| **TTS**（文字→语音） | CosyVoice / CosyVoice 2 | 阿里开源，零样本克隆 |
| **TTS** | ChatTTS | 国产开源，聊天场景 |
| **TTS** | ElevenLabs | 商用，效果顶级 |
| **Realtime 语音对话** | GPT-4o Realtime | 低延迟 < 300ms |

## 变体维度 6：行业 / 场景

### 行业模型

针对特定行业做了大量微调：

| 行业 | 代表模型 | 特点 |
|------|---------|------|
| **金融** | BloombergGPT、FinGPT、轩辕 | 财报分析、风险评估 |
| **医疗** | Med-PaLM 2、扁鹊（HuaTuo）、DISC-MedLLM | 临床问答、影像 |
| **法律** | ChatLaw、通义法睿、LawGPT | 法规检索、合同分析 |
| **代码** | Qwen3.6-Coder、DeepSeek-Coder-V3 | 已是细分方向 |
| **教育** | 智谱 EduAgent、作业帮系列 | K12 辅导 |

**经验**：通用模型（GPT-5.5 / Claude Opus 4.7 / DeepSeek V4）在大多数场景已足够，**行业模型只在合规要求严格、专业知识密度极高的场景才有必要**（如三甲医院、金融监管）。

### 区域 / 语言变体

- **国产模型**：Qwen3.x、DeepSeek V4、GLM-5.1、Kimi K2.6、文心 5.1、豆包 2.0——中文优化 + 数据合规
- **多语言模型**：Llama 4 Scout（100+ 语言）、Qwen3.6（201 语言）、Mixtral 8x22B
- **区域合规模型**：Falcon 3（阿联酋）、Jais 2（阿拉伯语）

> 国内业务**优先选国产开源模型**——中文能力强、推理速度快、数据不出境、可商用。

## 变体选型决策框架

| 你的需求 | 推荐变体 | 代表 |
|---------|---------|------|
| 普通对话 / Agent 工具调用 | 通用大模型 | Sonnet / GPT-4o / DeepSeek V3 |
| IDE 代码补全（< 100ms） | **Code 专用小模型** | Codestral 7B / Qwen-Coder 1.5B |
| 数学 / 逻辑难题 | 推理模型 | o1 / R1 / Gemini Thinking |
| RAG 检索（向量召回） | **Embedding 模型** | BGE-M3 / text-embedding-3 |
| RAG 精排 | **Rerank 模型** | BGE-reranker-v2-m3 / Cohere Rerank |
| 多模态（图片理解） | Vision 模型 | GPT-4V / Claude / Gemini |
| 离线部署 + 性能敏感 | **4-bit 量化** | Q4_K_M GGUF |
| 离线部署 + 能力要求高 | **蒸馏版** | Distill-Qwen-7B |
| 消费级 GPU 跑 70B | **AWQ 4-bit** | AutoAWQ + offload |
| 行业知识（金融 / 医疗） | 行业微调模型 | BloombergGPT / 扁鹊 |
| 语音转文字 | **STT 模型** | Whisper / Paraformer |
| 文字转语音 | **TTS 模型** | CosyVoice / ElevenLabs |
| Agent 开发用开源模型 | **Instruct / Chat 版本** | Llama-3-8B-Instruct / Qwen2.5-7B-Instruct |

## 总结

- **变体不是营销概念，是 LLM 生态分工协作的必然产物**——6 大维度（训练阶段 / 能力专精 / 部署形态 / 蒸馏 / 专用家族 / 行业场景）共同构成完整生态
- **训练阶段变体**：Base / Instruct / Chat / Reasoning 4 种，**Agent 开发永远选 Instruct 或 Chat**
- **能力专精变体**：Code / Math / Vision / Function Calling 各有专门模型，**按需选型，不要用大炮打蚊子**
- **部署形态变体**：**量化是消费级 GPU 跑大模型的关键**（Q4_K_M 是标配），合并和剪枝是进阶手段
- **蒸馏变体**：用教师监督让小模型"继承"大模型能力，实现"小而精"——来源特殊，既非纯后训练也非纯后处理
- **专用模型家族**：Embedding / Rerank / 语音模型是 **RAG / Agent / 多模态系统的组件库**，混用会出大问题
- **行业变体**：通用模型通常已够用，**只在合规要求严、专业密度高的场景**用行业模型
- **变体选型核心**：先明确需求（在线 / 离线？延迟 / 质量？通用 / 专用？），再选变体；不要先选最大模型再想怎么压缩

> 了解了变体全景之后，下一篇介绍一类特殊的变体——它们不是靠“更大”来变强，而是靠“想得更久”。请前往 [推理模型专题](./05-deep-thinking-and-reasoning.md)。
