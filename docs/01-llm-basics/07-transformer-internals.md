你好，我是江小湖。[NLP 到 Transformer](./02-nlp-to-transformer.md)里我们知道了 Transformer 是 LLM 的引擎，但它到底是怎么运转的？这篇把引擎拆开，看看里面的齿轮是怎么咬合的。

作为应用开发者，你不需要手写 Transformer，也不需要推导数学公式。但你需要建立对它内部机制的**直觉**，因为 Transformer 的每一个设计选择，都直接影响了你开发 Agent 时遇到的实际问题。

## 一个完整的 Transformer Block

先看全景，再拆零件。

**一个 Transformer Block 就是一个"理解+变换"单元**，LLM 内部堆叠了几十甚至上百个这样的 Block。每个 Block 的结构固定：

```
输入向量
    ↓
[多头自注意力]  ← 决定"看向哪里"
    ↓
[Add & Norm]    ← 残差连接 + 归一化（防崩溃）
    ↓
[前馈网络 FFN]   ← 决定"怎么处理看到的信息"
    ↓
[Add & Norm]
    ↓
输出向量 → 传给下一个 Block
```

  <br/>

</p>

**两个关键角色的分工**：多头自注意力决定**看向哪里**（哪些 Token 之间有关联），前馈网络（FFN）决定**怎么处理**这些关联信息。它们通过残差连接（Add）串联，保证信息不会在层层传递中丢失。

一个 GPT-4 级别的模型大约有 100+ 个这样的 Block。输入文本经过 Block 1 → Block 2 → ... → Block 100，每一层都在上一层的理解基础上进一步提炼语义。这就是为什么大模型理解能力更强——它有更多的"消化层次"。

## 自注意力的计算过程

[NLP 到 Transformer](./02-nlp-to-transformer.md)里已经用鸡尾酒会的类比解释了自注意力的直觉：每个 Token 都会去看其他所有 Token，计算相关性。现在我们拆开看它**具体怎么算**。

### Q/K/V：图书馆查资料

每个 Token 进入注意力层时，会被转换成三个向量：

- **Q（Query/查询）**：当前 Token 想问的问题——"谁能帮我确定我的含义？"
- **K（Key/键）**：每个 Token 贴在自己身上的标签——"我含有这类信息"
- **V（Value/值）**：每个 Token 实际携带的信息内容

拿 Q 去和所有 K 匹配，匹配度越高就越关注那个 Token 的 V。

### 数值计算示例

拿一个极简句子 `"我喜欢 AI"` 来走一遍（假设每个 Token 的 Q/K/V 已经是 2 维向量）：

```
Token   Q向量        K向量        V向量
我喜欢  [1.0, 0.2]  [0.8, 0.1]  [0.9, 0.3]
我      [0.5, 0.8]  [0.3, 0.9]  [0.4, 0.7]
喜欢    [0.2, 1.0]  [0.1, 0.8]  [0.1, 0.9]
AI      [0.8, 0.1]  [1.0, 0.2]  [0.9, 0.1]
```

第 1 步：计算"我喜欢"对所有 Token 的注意力分数——Q 和 K 的点积：

```
注意力分数 = Q("我喜欢") · K(各Token)
  我喜欢 → 我喜欢: 1.0×0.8 + 0.2×0.1 = 0.82
  我喜欢 → 我:     1.0×0.3 + 0.2×0.9 = 0.48
  我喜欢 → 喜欢:   1.0×0.1 + 0.2×0.8 = 0.26
  我喜欢 → AI:     1.0×1.0 + 0.2×0.2 = 1.04
```

第 2 步：分数经过 softmax 变成权重（总和为 1）：

```
权重 = softmax([0.82, 0.48, 0.26, 1.04])
  → 约为 [0.24, 0.14, 0.10, 0.52]
  "我喜欢"对"AI"的注意力最高（0.52）
```

第 3 步：用权重混合所有 Token 的 V，得到"我喜欢"的新表示：

```
新向量 = 0.24×V("我喜欢") + 0.14×V("我") + 0.10×V("喜欢") + 0.52×V("AI")
```

**关键理解**：经过注意力层后，"我喜欢"这个 Token 的向量已经不再是孤立的——它融合了对"AI"的高度关注（权重 0.52），变成了一个**"喜欢 AI"的复合语义向量**。

实际 LLM 中，Q/K/V 的维度通常是 64-128 维，Token 数量可达数万，但计算逻辑和这个 2 维示例完全相同。

## 多头注意力

自注意力有一个问题：**只用一组 Q/K/V，每个 Token 只能建立一种关联**。

但语言中的关联是多维度的。同一个词"苹果"：
- 在语法层面，需要和动词"吃"关联（主谓关系）
- 在语义层面，需要和"手机""乔布斯"关联（歧义消解）
- 在位置层面，需要和前后相邻词关联（短语结构）

**多头注意力（Multi-Head Attention）** 就是让每个 Block 同时运行多组独立的注意力计算，每组用自己的 Q/K/V，从不同角度捕捉关联。GPT-4 级别的模型通常有 96 个注意力头。

类比：**多头注意力就像 96 个侦探同时调查同一个案件**，每个侦探关注不同的线索——有人盯语法关系，有人盯语义相似度，有人盯上下文距离。最后把 96 个侦探的发现汇总拼接，就得到了对文本的全方位理解。

这比单头注意力的理解深度大得多。这也是 Transformer 为什么强大——不是因为它能"看所有词"，而是因为它能从**多个角度**同时看所有词。

## 因果掩码

自回归生成（[LLM 概述](./01-llm-overview.md)已讲过：预测→拼接→再预测的循环）有一个核心约束：**生成第 N 个 Token 时，模型只能看到前 N-1 个 Token，不能偷看后面的内容。**

否则训练时模型就会"作弊"——看到答案后再写答案，考试就失去了意义。

**因果掩码（Causal Mask）** 就是实现这个约束的机制。在计算注意力分数时，用一个矩阵把"未来"的分数全部置零：

```
对于 "中国的首都是北京" 6个Token的注意力矩阵：

        中国  的  首  都  是  北京
中国    [✓   ✓   ✗   ✗   ✗   ✗ ]   ← "中国"只能看自己
的      [✓   ✓   ✗   ✗   ✗   ✗ ]   ← "的"能看"中国"和自身
首      [✓   ✓   ✓   ✗   ✗   ✗ ]   ← "首"能看前3个
都      [✓   ✓   ✓   ✓   ✗   ✗ ]
是      [✓   ✓   ✓   ✓   ✓   ✗ ]
北京    [✓   ✓   ✓   ✓   ✓   ✓ ]   ← 最后一个Token能看全部

✓ = 允许关注    ✗ = 掩码屏蔽（置零）
```

这个下三角掩码确保了：每个 Token 的理解只基于它左边的上下文，绝不"偷看"右边还没生成的 Token。这就是 **Decoder-only** 架构的核心特征。

## 位置编码

Transformer 一次性处理所有 Token，这带来了一个天然缺陷：**它不知道词的顺序**。

"狗咬人"和"人咬狗"，对 Transformer 来说输入的词完全一样。如果没有任何顺序标记，它认为这两句话是一个意思。

**位置编码（Positional Encoding）** 的作用就是在每个 Token 的 Embedding 向量上叠加一个"位置信号"，让模型知道"这个词在第几个位置"。

### RoPE：旋转位置编码

现代 LLM（GPT-4、Llama、Qwen 等）几乎都用 **RoPE（旋转位置编码）**。它的核心思想很优雅：

> 把每个 Token 的向量在 2D 子空间上**旋转**一个角度，角度的大小和位置编号成正比。

这样一来，两个 Token 的注意力分数（Q·K 点积）会自然地包含它们**相对距离**的信息——位置 m 和位置 n 的 Token，它们的点积受 (m-n) 的影响，而不是绝对位置。这就是 RoPE 能支持超长上下文的原因：模型关注的是"你离我多远"，不是"你在第几行"。

对比早期 Transformer 用的绝对位置编码（直接加位置编号），RoPE 的相对位置特性让模型对长文本的理解更稳定——开头和结尾的词之间的关系，不会因为文本长度变化而失真。

## KV Cache 与推理成本

理解了因果掩码和自注意力的计算流程，就能看懂 LLM 推理的一个核心优化：**KV Cache**。

### 推理时的计算浪费

自回归生成每一步都要重新计算注意力。假设模型已经生成了 100 个 Token，现在要生成第 101 个：

- 第 101 个 Token 的 Q 需要和**前 100 个 Token 的 K** 全部做点积
- 但前 100 个 Token 的 K 和 V **每一步都没变**——它们是已经确定的历史，不会因为新 Token 的出现而改变

如果不做优化，每一步都要把前 100 个 Token 的 K/V 重新算一遍。生成 1000 个 Token 的文本，就要重复计算 1000 次越来越长的 K/V。这就是为什么**生成越长越慢**。

### KV Cache 的原理

**KV Cache** 的做法很简单：把已经算过的 K 和 V **缓存下来**，新 Token 只需要：
1. 计算自己的 Q/K/V
2. 从缓存中读取前 N-1 个 Token 的 K/V
3. 用自己的 Q 和缓存的 K/V 计算注意力

每一步的计算量从 O(N²) 降低到 O(N)，推理速度大幅提升。

**对 Agent 开发者的实际影响**：

- **为什么 ChatGPT 回复越写越慢**：随着生成 Token 增加，虽然 KV Cache 避免了重复计算 K/V，但每步的注意力计算量仍在增长（新 Q 要和越来越多的历史 K 做点积）
- **为什么上下文窗口扩展很贵**：1M tokens 的上下文意味着 KV Cache 要存储 1M×96heads×128dim 的浮点数，内存消耗巨大。这就是为什么长上下文的 API 价格显著更高
- **为什么多轮对话越来越慢**：每一轮对话的历史消息都进入 KV Cache，对话越长，每步推理的注意力计算量就越大

## 总结

这篇拆开了 Transformer 的内部机制，关键要点：

- **Transformer Block** = 多头自注意力（看向哪里）+ 前馈网络（怎么处理），堆叠几十到上百层，层层提炼语义
- **自注意力** 通过 Q·K 点积计算关联权重，再用权重混合 V，让每个 Token 融合上下文信息
- **多头注意力** 从多个角度同时捕捉关联，是 Transformer 强于 RNN 的核心原因
- **因果掩码** 阻止模型偷看未来 Token，是自回归生成的纪律保障
- **RoPE 位置编码** 用旋转角度编码相对位置，是长上下文支持的基础
- **KV Cache** 缓存历史 K/V 避免重复计算，但注意力计算量仍随上下文长度增长——这直接解释了 Agent 开发中的成本和性能问题
