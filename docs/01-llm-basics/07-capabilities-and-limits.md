## 概述

在开发 Agent 之前，你必须清楚地知道 LLM 的边界在哪里。

把 LLM 想象成一个极其聪明、读过世界上所有书、但被关在一个没有网的黑屋子里、且偶尔会自信地胡说八道的人。**Agent 工程的本质，就是用代码把这个人的优点放大，用工具把他的缺点补齐。**

## LLM 能做什么

现代大语言模型（如 GPT-4o、Claude 3.5 Sonnet）具备以下核心能力：

1. **文本理解与生成**：这是最基础的能力。它可以做摘要、翻译、润色、扩写，并且能理解极其复杂的长文本。
2. **逻辑推理（Reasoning）**：它能解决逻辑谜题、进行常识推理。特别是最新的推理模型（如 o1、DeepSeek-R1），通过在生成前进行内部的"思维链（Chain of Thought）"计算，推理能力有了质的飞跃。
3. **代码生成与分析**：LLM 在海量 GitHub 代码上训练过，能熟练编写各种语言的代码、寻找 Bug、写单元测试。这是目前 LLM 创造商业价值最大的领域之一。
4. **上下文学习（In-context Learning）**：你不需要重新训练模型，只要在 Prompt 里给它几个例子（Few-shot），它就能学会一种新的输出格式或处理逻辑。
5. **指令遵循与角色扮演**：通过 System Prompt，你可以让它扮演成一个严厉的代码审查员、一个温柔的心理医生，或者一个只输出 JSON 格式的 API 接口。

## LLM 的核心局限

了解局限比了解能力更重要，因为你在开发中遇到的大部分 Bug 都来源于此。

### 1. 幻觉（Hallucination）
**现象**：模型会极其自信地编造不存在的事实、伪造论文引用、或者调用不存在的 API 函数。
**原因**：它的底层原理是"预测下一个词"，而不是"检索数据库"。它在努力生成一段"看起来很合理"的文本，而不是"绝对真实"的文本。

### 2. 知识截止（Knowledge Cutoff）
**现象**：如果你问它昨天发生的新闻，它要么说不知道，要么胡编乱造。
**原因**：模型的知识停留在它完成预训练的那一天。训练一个大模型需要几个月，所以它的知识永远是滞后的。

### 3. 严格数学与精确逻辑的弱点
**现象**：让它算 `34598 * 23489`，它可能会给出一个看起来很像正确答案的错误数字。让它统计一段话里有几个字母 "r"，它经常数错。
**原因**：Token 机制导致它看不到单个字母（它看到的是 subword）；自回归机制导致它无法像计算器那样进行精确的符号运算。

### 4. 上下文窗口限制与"迷失在中间"
**现象**：虽然现在的模型号称支持 200K 甚至 2M 的上下文，但如果你塞给它一本厚书，然后问它书中间的一个细节，它很可能会忽略掉。
**原因**：这被称为 **Lost in the middle（迷失在中间）** 现象。模型对 Prompt 开头和结尾的信息注意力最集中，对中间的信息容易忽略。

## 对应用开发的影响

理解了上述局限，你就能明白为什么现在的 Agent 架构长这样：

| LLM 的局限 | Agent 工程的解决方案 |
|-----------|-------------------|
| **知识截止** | **RAG（检索增强生成）**：在提问前，先去数据库或搜索引擎找资料，把资料塞进 Prompt 里让它参考。 |
| **数学/逻辑弱点** | **工具调用（Tool Use / Function Calling）**：给它一个计算器工具或 Python 执行环境，让它写代码来算，而不是自己算。 |
| **幻觉** | **Grounding（事实锚定）**：要求它在回答时必须引用 RAG 提供的资料片段；或者引入另一个 LLM 来做交叉验证（Eval）。 |
| **上下文限制** | **记忆管理（Memory）**：不能把所有历史对话都塞进去，需要做向量检索、对话摘要，只保留最相关的上下文。 |
| **复杂任务容易出错** | **Agent 框架（如 LangGraph）**：把大任务拆解成小步骤（SOP），每一步只让 LLM 做一个简单的决策，通过状态机控制流程。 |

**总结**：不要把 LLM 当成全知全能的神。把它当成一个**强大的文本处理器和决策引擎**，然后用传统的软件工程方法（API、数据库、逻辑控制）去武装它。这就是 Agent 开发的核心心法。
