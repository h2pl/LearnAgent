你好，我是江小湖。在第 5 章中，你用纯 Python 手写了一个最小 Agent——几十行代码就实现了 ReAct 循环。那为什么还需要框架？

因为 demo 和生产系统之间有一道鸿沟：并发控制、状态持久化、错误恢复、人机协作、多 Agent 协调……这些"非核心逻辑"的代码量可能是核心逻辑的 10 倍以上。框架的价值，就是帮你把这些通用能力标准化，让你专注于业务逻辑。

读完本文，你将理解：框架解决了什么问题、主流框架各自的特点、如何根据项目需求做出选择。

## 为什么需要框架

先回顾一下第 5 章的最小 Agent：

```python
# 最小 ReAct Agent（伪代码）
while True:
    response = llm.invoke(messages)
    if response.tool_calls:
        result = execute_tool(response.tool_calls[0])
        messages.append({"role": "tool", "content": result})
    else:
        return response.content
```

几十行代码，简洁明了。但当你尝试把它变成一个可用的产品时，会遇到一系列问题：

### 复杂流程管理

简单的循环很快就不够用了。你可能需要：

- **条件分支**：根据用户意图走不同的处理路径
- **并行执行**：同时调用多个工具，汇总结果
- **循环与重试**：某个步骤失败后自动重试，最多重试 3 次
- **子任务拆解**：将复杂任务分解为多个步骤，逐步执行

用纯 Python 写这些逻辑，代码会迅速膨胀为几百行的 if-else 嵌套，难以维护。

### 状态管理

Agent 运行过程中会产生大量状态：

- 对话历史
- 工具调用结果
- 中间推理步骤
- 当前执行进度

这些状态需要在内存中维护，重启后需要恢复，多用户并发时需要隔离。手写这些逻辑既繁琐又容易出错。

### 错误处理

生产环境中，错误无处不在：

- LLM API 超时或返回错误
- 工具调用失败（网络问题、权限不足）
- 模型输出格式不符合预期
- 循环卡死（模型反复调用同一个无效工具）

每个错误都需要对应的处理策略：重试、降级、熔断、人工介入。手写这些逻辑会让代码变得臃肿且难以测试。

### 可观测性

你需要知道：

- Agent 执行了哪些步骤？
- 每个步骤花了多长时间？
- 调用了哪些工具？结果是什么？
- 哪里出错了？错误原因是什么？

没有框架支持，你需要自己实现日志、追踪、指标采集——这些是后端工程的基本功，但和 Agent 业务逻辑无关。

## 框架提供了什么

一个成熟的 Agent 框架通常提供以下能力：

| 能力 | 说明 | 手写成本 |
|------|------|----------|
| **流程编排** | 用图/状态机定义 Agent 的执行流程 | 高 |
| **状态管理** | 自动维护对话历史、中间结果、执行状态 | 中 |
| **工具集成** | 标准化的工具定义和调用机制 | 低 |
| **错误处理** | 重试、降级、超时控制 | 中 |
| **持久化** | 状态保存到磁盘/数据库，支持中断恢复 | 高 |
| **Human-in-the-loop** | 在关键步骤暂停，等待人类审批 | 高 |
| **可观测性** | 内置日志、追踪、回调 | 中 |
| **多 Agent 支持** | Agent 之间的通信和协调 | 非常高 |

**核心价值**：框架把"怎么做"（基础设施）和"做什么"（业务逻辑）分离。你只需要定义 Agent 的流程和决策逻辑，框架负责执行、状态管理、错误恢复等通用问题。

## 主流框架概览

截至 2025 年，Agent 框架领域已经相当成熟。以下是几个最具影响力的框架：

### LangGraph

**出品方**：LangChain 团队

**核心理念**：用**有向图**（状态图）定义 Agent 流程。节点是处理步骤，边是流转条件，状态在节点间传递。

**适合场景**：

- 流程复杂、需要精细控制的 Agent
- 需要中断恢复、Human-in-the-loop 的场景
- 对可观测性要求高的生产系统

**典型用户**：企业级 Agent 应用、需要复杂工作流的团队

### CrewAI

**出品方**：CrewAI 团队

**核心理念**：用**角色扮演**的方式组织多个 Agent 协作。每个 Agent 有明确的角色、目标和工具，通过任务分配来协作。

**适合场景**：

- 多 Agent 协作（如研究团队、内容团队）
- 任务可以自然拆分为多个角色
- 快速原型验证

**典型用户**：内容生成、研究分析、需要多角色协作的场景

### AutoGen

**出品方**：微软

**核心理念**：用**对话**驱动 Agent 协作。多个 Agent 通过消息传递来完成任务，支持代码执行和人机协作。

**适合场景**：

- 需要代码执行能力的场景
- 多 Agent 对话式协作
- 研究和实验性质的项目

**典型用户**：研究人员、数据科学家

### Dify

**出品方**：Dify.AI

**核心理念**：**低代码**平台，用可视化界面构建 Agent 应用。支持 RAG、工具调用、工作流编排。

**适合场景**：

- 非技术团队快速搭建 Agent 应用
- 需要快速验证商业想法
- 对代码控制要求不高的场景

**典型用户**：产品经理、业务团队、初创公司

### Semantic Kernel

**出品方**：微软

**核心理念**：面向**企业级**的 AI 编排框架，深度集成 Azure 生态。支持 C# 和 Python。

**适合场景**：

- 微软技术栈的企业
- 需要与 Azure AI 服务深度集成
- 对企业级安全和合规有要求

**典型用户**：微软生态的企业开发团队

## 框架对比

| 维度 | LangGraph | CrewAI | AutoGen | Dify | Semantic Kernel |
|------|-----------|--------|---------|------|-----------------|
| **编程模型** | 有向图（状态机） | 角色+任务 | 对话驱动 | 可视化配置 | 插件+规划器 |
| **学习曲线** | 中等 | 低 | 中等 | 非常低 | 中等 |
| **灵活性** | 非常高 | 中等 | 高 | 低 | 高 |
| **多 Agent** | 支持 | 原生支持 | 原生支持 | 有限 | 支持 |
| **HITL** | 原生支持 | 支持 | 支持 | 支持 | 支持 |
| **持久化** | 原生支持 | 有限 | 有限 | 内置 | Azure 集成 |
| **可观测性** | LangSmith 集成 | 基础日志 | 基础日志 | 内置 | Azure Monitor |
| **社区活跃度** | 高 | 高 | 中等 | 非常高 | 中等 |
| **生产就绪度** | 高 | 中等 | 低 | 高 | 高 |

### 一句话总结每个框架

- **LangGraph**：最灵活，适合复杂流程，但需要理解状态图概念
- **CrewAI**：最容易上手，适合多 Agent 协作，但定制能力有限
- **AutoGen**：最适合代码执行场景，但生产就绪度较低
- **Dify**：最适合非技术团队，但灵活性最低
- **Semantic Kernel**：最适合微软生态企业，但锁定 Azure

## 选型决策

### 第一步：明确你的场景

| 你的需求 | 推荐框架 |
|----------|----------|
| 复杂流程、精细控制 | LangGraph |
| 多 Agent 角色协作 | CrewAI |
| 快速原型、非技术团队 | Dify |
| 代码执行、研究实验 | AutoGen |
| 微软/Azure 企业栈 | Semantic Kernel |
| 简单循环、Agent 逻辑不复杂 | 不需要框架 |

### 第二步：评估技术约束

| 约束 | 影响 |
|------|------|
| **团队技术栈** | Python 为主选 LangGraph/CrewAI；.NET 选 Semantic Kernel |
| **部署环境** | 云端优先选 Dify；本地部署选 LangGraph |
| **合规要求** | 数据不能外传选可本地部署的框架 |
| **性能要求** | 高并发选 LangGraph（支持异步和流式） |

### 第三步：考虑长期维护

- **社区活跃度**：社区活跃意味着 bug 修复快、文档完善、遇到问题容易找到解决方案
- **API 稳定性**：频繁 breaking change 会增加维护成本
- **可替代性**：框架是否容易替换？如果需要迁移，成本有多高？

**建议**：从 LangGraph 开始。它是最通用的选择，覆盖了绝大多数场景。如果项目明确需要多 Agent 协作，再考虑 CrewAI。

## 什么时候不需要框架

框架不是万能的。以下场景，手写可能更好：

### Agent 逻辑非常简单

如果你的 Agent 就是一个简单的循环——接收问题、调用工具、返回结果——那几十行 Python 就够了。引入框架反而增加了不必要的复杂性。

### 性能极致敏感

框架会引入额外的抽象层，带来一定的性能开销。如果你的场景对延迟和吞吐量有极致要求（比如每秒处理上千个请求），手写可能是更好的选择。

### 学习成本不值得

如果你的团队不熟悉某个框架，花几天时间学习框架的 API 和概念，可能不如直接写代码来得快。特别是对于小项目或短期项目。

### 需要完全控制

有些场景需要对执行流程有完全的控制——比如自定义的调度算法、特殊的并发模型。框架的抽象可能反而限制了你的灵活性。

**经验法则**：如果你的 Agent 代码超过 500 行，或者需要多人协作开发，或者需要处理错误恢复和状态持久化，那就值得考虑用框架。

## 总结

- **框架的核心价值**：把"怎么做"（基础设施）和"做什么"（业务逻辑）分离，让你专注于 Agent 的业务逻辑
- **框架提供了什么**：流程编排、状态管理、错误处理、持久化、HITL、可观测性
- **主流框架**：LangGraph（最灵活）、CrewAI（最易上手）、AutoGen（代码执行）、Dify（低代码）、Semantic Kernel（微软生态）
- **选型建议**：从 LangGraph 开始，它是通用性最好的选择
- **什么时候不用**：逻辑简单、极致性能、学习成本不值得、需要完全控制
