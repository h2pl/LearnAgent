# 两条技术路线：理解 vs 生成

多模态看起来是一个概念，但底层是两种完全不同的技术路线。**理解侧（Understanding）** 和 **生成侧（Generation）** 从架构到训练到计费都不是一回事。

这个区分对开发者的实际影响比你想象的大：你在用 DALL-E 生成图片时走的是扩散模型链路，在用 ChatGPT 分析图片时走的是 Transformer 链路。它们之间比你想象的独立——独立训练、独立部署、独立计费。

## 理解侧：让模型"看懂"和"听懂"

理解侧的任务是：输入图像/音频/视频，输出文字或结构化信息。典型产品：ChatGPT 图片问答、Gemini 视频理解、语音转文字。

**架构：Transformer + 感知编码器**

```
图像 → ViT 编码器 → patch 向量序列 ┐
音频 → 音频编码器 → 帧向量序列    → Transformer（自注意力）→ 文字/结构化输出
文本 → Tokenizer → token 向量序列  ┘
```

核心是你已经熟悉的 Transformer，前面加一个编码器把非文本输入转成向量序列。ViT（Vision Transformer）把图片切成 patch 编码，Whisper 把音频转成序列编码，然后和文本 token 一起过 Transformer。

**推理方式**：一次前向传播。输入送到模型，计算一次，出结果。典型延迟 0.5-3 秒。

**计费**：按 token 计费。一张图 ≈ 1000-2000 token，一段音频按帧数折算。这是为什么多模态理解 API 比纯文本贵得多——一张输入图片的"token 成本"可能相当于一整段对话的文本 token。

**能力边界**：理解侧擅长"识别"和"分析"类任务，不擅长"创造"。Claude 能准确识别图片中的 UI 元素，但让它生成一张一样的 UI 界面——它做不了（那是生成侧的事）。

## 生成侧：让模型"创造"视觉和听觉内容

生成侧的任务是：输入文字（或图片），输出图像/视频/音乐。典型产品：Midjourney、DALL-E 3、Flux、Sora、Suno、Kling。

**架构：扩散模型 + 条件引导**

```
文字描述 → 文本编码器（CLIP/T5）→ 条件向量
↓
纯噪声 → 去噪网络（UNet/DiT）← 条件引导（20-50 步迭代去噪）
↓
VAE 解码器 → 清晰图像/视频/音频
```

注意这个过程完全是反直觉的：**从一个纯噪声开始，逐步去掉噪声，最终得到清晰的内容。** 这不是"从无到有"，而是"从乱到治"。每一步都在根据文字条件微调当前输出，让它更匹配文字描述。

**推理方式**：20-50 步迭代。每步一次小计算，串行执行。典型延迟 3-30 秒（图片）、1-5 分钟（视频）。这是为什么文生图总是很慢——不是模型大，是它需要跑几十步才能从噪声中还原出图像。

**计费**：按张数/时长/步数计费。Midjourney 按生成次数收费，DALL-E 按图片张数收费——和 token 消耗无关。

**能力边界**：生成侧擅长"创造"类任务，不适合"分析"类任务。Flux 能生成一张"戴帽子的猫坐在打字机前"的图片，但让它识别图片里有什么——做不了。

## 为什么这个区分很重要

你可能会想：GPT-5.5 不是既能分析图片又能生成图片吗？它应该是统一的吧？

不是。GPT-5.5 的"统一"是在 API 层面的——你通过同一个 API 传入"分析这张图"和"生成一张图"的请求，但底层调用的是两个完全不同的模型：

```
你: "分析这张图有什么问题"
    ↓
GPT-5.5 API → 路由到理解侧（Transformer）→ 分析结果

你: "生成一张夕阳下的海滩图片"
    ↓
GPT-5.5 API → 路由到生成侧（扩散模型）→ 图片
```

理解侧用 Transformer，生成侧用扩散模型。两者独立训练、独立部署、独立更新、独立计费。API 统一只是让开发者不用维护两套 SDK。

这对选型的影响：

**文生图/视频/音乐场景，只看生成模型。** Midjourney 的审美、Flux 的控制精度、Sora 的视频一致性——这些是各自训练出来的，和理解侧模型的能力没有关系。理解侧模型在 Arena 视觉榜排第几，不影响生成侧的质量。

**分析/理解场景，只看理解模型。** Claude 在结构化输出上的优势、GPT-5.5 在推理链上的能力、Gemini 在超长上下文上的优势——这些是 Transformer 的能力。扩散模型做不了这些任务。

**两套预算体系。** 理解侧按 token 优化（降分辨率、裁剪图片、限音频时长），生成侧按步数优化（减少采样步数、用蒸馏模型）。优化策略完全不同。

## 生成侧的三个核心挑战

### 控制精度

"一只戴帽子的橘猫"——指令很简单，但扩散模型怎么确保生成的内容精确匹配？条件引导的质量决定了控制精度。ControlNet、LoRA 等技术在 2025-2026 年大幅提升了控制能力，让你可以精确定位画面中每个元素的位置。

### 速度

扩散模型需要 20-50 步迭代才能从噪声还原出图像。每一步是一轮 UNet/DiT 计算。速度优化从三个方向突破：
- **少跑几步**：采样算法优化（DPM++、LCM），把需要的步数从 50 降到 3-5
- **每步跑快**：模型蒸馏、量化，减少每步的计算量
- **并行处理**：视频生成时可以并行处理多个帧，但帧间一致性做不好

### 视频生成的时间一致性

这是 2026 年最活跃的研究方向之一。单帧生成已经很好，但相邻帧之间的物体一致性（人不能突然消失、背景不能突变）是视频生成的核心难点。Sora 用时空 patch 把"帧"的概念泛化了，其他方案（Kling、Runway）则是在扩散过程的每一步加入时序约束。

## 两条路线的全面对比

| 维度 | 理解侧 | 生成侧 |
|------|--------|--------|
| 核心架构 | Transformer（自注意力） | 扩散模型（迭代去噪） |
| 信息流 | 多模态输入 → 文字输出 | 文字/图片输入 → 图像/视频/音频输出 |
| 推理方式 | 一次前向传播 | 20-50 步迭代 |
| 典型延迟 | 0.5-3 秒 | 3-30 秒（图）/ 1-5 分钟（视频） |
| 计费方式 | 按 token | 按张数/时长/步数 |
| 优化方向 | 降 token 消耗 | 降采样步数 |
| 代表产品 | ChatGPT 看图、Gemini 视频分析 | Midjourney、Sora、Suno |
| 当前瓶颈 | 跨模态对齐精度 | 生成速度 + 控制精度 |
