# 两条技术路线：理解 vs 生成

在深入多模态的具体机制之前，有一个根本性的区分必须先搞清楚——多模态 AI 其实是两条完全不同的技术路线拼在一起的。

**理解侧**让模型"看懂"和"听懂"，输入图像/音频，输出文字或结构化信息。**生成侧**让模型"创造"，输入文字描述，输出图像/视频。它们使用的架构、训练方式、推理过程截然不同。

## 理解侧：Transformer + 感知编码器

理解侧的代表是 GPT-5.5、Claude Opus 4.8、Gemini 2.5 Pro 这些模型。它们的本质是**大语言模型加了一个"感知前端"**：

```
图像/音频/视频 → 感知编码器（ViT / Audio Encoder）→ 向量序列
↓
文字输入 → Tokenizer → 向量序列 → LLM（Transformer）
↓
文字/结构化输出
```

核心架构和你已经熟悉的 LLM 是同一个 Transformer。只不过在纯文本 Transformer 前面加了一个视觉编码器（比如 Vision Transformer）或音频编码器，让模型能"看到"和"听到"。

理解侧有三个核心机制：编码（把图片/音频变成向量）、融合（让不同模态的向量交互）、对齐（匹配跨模态的对应关系）。接下来的三篇会逐篇深入这三个机制。

## 生成侧：扩散模型 + 条件引导

生成侧的代表是 DALL-E 3、Stable Diffusion 3.5、Flux、Sora。它们用的架构完全不同——**扩散模型（Diffusion Model）**：

```
文字描述 → 文本编码器（CLIP / T5）→ 条件向量
↓
纯噪声 → 去噪网络（UNet / DiT）← 条件引导（逐步去噪 20-50 步）
↓
VAE 解码器 → 清晰图像/视频
```

核心是**迭代去噪**过程：从一个纯噪声图像开始，按照文字条件的引导，逐步去掉噪声，最终还原出清晰的图像。

生成侧的核心挑战是：怎么精确控制生成内容（条件引导）、怎么提高质量和速度（采样算法、模型蒸馏）、怎么扩展到视频（时间一致性）。这些会在[图像与视频生成](../04-multimodal-generation/01-image-and-video-generation.md)中系统讲解。

## 两条路线的关键区别

| 维度 | 理解侧 | 生成侧 |
|------|--------|--------|
| 核心架构 | Transformer（自注意力） | 扩散模型（迭代去噪） |
| 信息流 | 多模态输入 → 文字输出 | 文字输入 → 图像/视频输出 |
| 推理方式 | 一次前向传播（快） | 20-50 步迭代去噪（慢） |
| 典型模型 | GPT-5.5、Claude Opus 4.8、Gemini 2.5 Pro | DALL-E 3、Flux、Sora |
| 典型延迟 | 0.5-3 秒 | 3-30 秒（图片）/ 1-5 分钟（视频） |
| 训练数据 | 图文配对 + 指令微调 | 大规模图像/视频 + 文字描述 |
| 开发者接口 | Chat Completion API | Image Generation API |

**为什么要区分这两条路线？** 因为你在实际使用时，它们几乎总是分开的 API、分开的计费、分开的模型权重。即使 GPT-5.5 能同时做理解和生成，底层也是两套引擎在跑。把"多模态"当成一套统一技术来理解，会产生很多误判——比如以为理解了编码融合对齐就能搞定生成，其实完全不是那么回事。

## 趋势：API 层面的统一

2026 年有一个趋势：Gemini 通过 Imagen 3 把生成能力集成到了同一个 API 接口，GPT-5.5 也能在一次对话中混合调用理解和生成。API 层面正在走向统一，但底层引擎仍然是两套。

这个趋势对开发者的好处是接口简化，但选型时仍然需要清楚：理解任务选理解模型，生成任务选生成模型。
