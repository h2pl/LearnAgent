# 编码：从像素到向量

多模态处理的第一步，是把每种模态的原始数据转换成模型能计算的数字向量。这个过程叫做编码（Encoding）。

不同模态的编码方式差异很大。编码方式直接决定了信息密度、计算成本，以及后续跨模态处理的难度。

## 文本编码

文本编码是三种模态中最成熟的。文本通过 Tokenizer 切成 token 序列，每个 token 对应一个嵌入向量（embedding）：

```
"猫在睡觉" → Tokenizer → ["猫", "在", "睡觉"] → Embedding → [[0.23, -0.15, ...], [0.41, 0.02, ...], ...]
```

文本编码的特点：
- **语义密度高**。每个 token 都承载明确的语义信息，没有"冗余"。
- **表示紧凑**。100 个词的文本 → 100 个向量，效率很高。
- **离散性**。token 是离散符号，不像像素和波形那样连续。

## 图像编码

图像编码的方式完全不同。主流方法是**把图片切成小块（patch）**，每个小块单独经过 Vision Transformer（ViT）编码成一个向量：

```
224×224 图片 → 切成 14×14 = 196 个 patch → 每个 patch 编码成一个向量 → 196 个向量
```

一张 224×224 的中等分辨率图片，产生 196 个向量。高分辨率图片会更多。

图像编码的特点：
- **空间结构重要**。patch 之间的位置关系就是信息的一部分。ViT 通过位置编码保留了这种空间信息。
- **信息冗余大**。196 个 patch 中很多可能只是背景（比如天空、墙壁），不承载关键信息。
- **分辨率敏感**。分辨率越高，patch 越多，token 消耗成倍增长。

这就是为什么图像会消耗大量 token——一张图可能就占了几百个 token 的预算。缩小图片分辨率、裁剪无关区域，是控制成本最直接的方式。

## 音频编码

音频编码的常见做法是把原始波形转成**梅尔频谱图（Mel Spectrogram）**，然后用类似图像编码的方式处理：

```
音频波形 → 分帧 → FFT → Mel 滤波器 → 频谱图 → ViT/CNN 编码 → 向量序列
```

另一种方式（Whisper 类模型）直接把音频波形编码成向量序列，和文本 token 放在同一个空间里处理。

音频编码的特点：
- **时序连续**。声音信息在时间上展开，需要处理长序列。
- **频率和时间的二维结构**。频谱图本质上是一张"时间×频率"的图像。

## 为什么编码方式很重要

不同模态的编码方式决定了三件事：

**信息密度**。文本最紧凑，每个 token 都有意义。图像最冗余，大量 patch 是背景。这直接影响了 token 消耗和成本——一张图可能抵几百个文本 token。

**计算成本**。图像和音频编码需要专门的编码器（ViT、音频编码器），不像文本那样只需要一个 embedding lookup。编码器的计算量直接反映在 API 延迟上。

**跨模态对齐难度**。编码后不同模态的向量在维度、语义空间上都不同。"猫"的文字向量（从 embedding table 来的）和"猫"的图像向量（从 ViT 来的）长度可能不同、分布也可能不同。后续的融合和对齐就是解决这个问题——也就是接下来两篇的内容。

## 实践建议

- **缩小图片能省钱**。1024×1024 的图片比 512×512 多产生 4 倍 patch token。如果不需要精细细节，降低分辨率是最有效的成本优化。
- **裁剪比缩小更好**。保留感兴趣区域、裁掉无关背景，比整体缩小更有效——关键信息的分辨率没降，但总 token 数大幅减少。
- **音频长度决定成本**。音频处理的 token 消耗和时长成正比，15 秒的语音比 5 秒贵 3 倍。
