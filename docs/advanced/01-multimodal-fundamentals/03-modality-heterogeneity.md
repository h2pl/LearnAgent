# 模态异质性

多模态听起来简单——不就是"支持更多输入类型"吗？你给模型传图片、传音频，它处理，你拿结果。真正做过的人会告诉你不是这么回事。

不同模态的信息天然存在于完全不同的表示空间里，这种差异叫**模态异质性（Modality Heterogeneity）**。这是多模态技术最根本的挑战，不理解它，你就理解不了为什么多模态 API 这么贵、为什么有些组合成熟有些还在早期、为什么有的模型做图文理解很厉害但完全不能生成音乐。

## 四种模态的底层结构

### 文本：离散的符号系统

文本是人类定义的抽象符号系统。每个"词"是一个离散的 token，通过 tokenizer 切分、embedding 映射成向量。

文本编码的特点：每个 token 语义密度极高，没有冗余。100 个词的文本 ≈ 100 个 token，每个 token 都有明确意义。但文本的问题是表达天花板低——你没法用文字精确描述一张脸上的微表情、一个声音中的颤抖、一段视频中的气氛变化。

这正是文生图 Prompt 工程困难的原因：文字描述"夕阳下的沙滩"是两个 token 的事，但 DALL-E 需要用几十步去噪算法把这个语义还原成像素级的视觉呈现——文本和图像之间的信息密度差异巨大，存在一个"语义→像素"的信息补全过程。

### 图像：空间的像素矩阵

一张 1024×1024 的图片是 1024×1024×3 = 314 万个数字。每个数 0-255，表示颜色强度。

图像和文本的根本区别：**空间结构就是信息本身**。"猫在图片的中央偏左"这种信息不是"描述"出来的，而是直接编码在像素排列中的。你移动一个像素的位置，画面内容可能从"猫"变成"扭曲的色块"——这和文本完全不同（调换两个无关词的顺序可能不影响语义）。

图像的另一个问题：信息冗余极大。一张蓝色天空背景 + 一只猫的照片，70% 的像素在描述"不同程度的蓝色"，但模型必须处理所有像素才能找到猫。这直接解释了为什么多模态 API 接收一张图的成本相当于几百个文本 token——模型需要计算的不只是"内容"，还有大量冗余信息。

### 音频：时间的波形

音频不像文本和图像那么直观。一段 16kHz 采样的语音，每秒 16000 个采样点，每个采样点是对应时刻的声压值。

音频最特别的地方：**信息在时间维度上展开，且多通道并存。** 同一段音频中包含语音内容（说了什么）、说话人身份（谁说的）、情绪状态（语气）、环境信息（在哪里录的）——这些全部编码在同一个波形中。这比图像难处理得多，图像中的人物和背景至少在空间上是可分离的。

文生音乐（Suno/Udio）的技术难点就在这里：生成的音频需要同时保证节奏正确、旋律和谐、音色自然、多乐器协调——所有东西都在同一段波形中存在。

不同模型处理音频的方式不同：
- **Whisper 类**：把音频编码成向量序列，和文本 token 放在同一空间处理，专做语音识别
- **Suno/Udio 类**：用扩散模型 + 音频编码器生成完整波形，专做音乐和音效生成
- **Realtime API 类**：流式处理音频帧，在理解侧模型和音频编码之间做实时转换

### 视频：空间叠时间

视频是最复杂的。30fps、1 分钟 = 1800 帧图像，加上帧间的时序关系。

视频不仅是"很多张图片"——帧之间的运动模式、事件序列、物体的连续变化，这些时序信息让视频理解的复杂度比图像高一个数量级。Sora 生成视频时最困难的是保持帧间一致性（画面中的人物不能突然消失或变形），这需要理解每一帧的内容，还要理解帧与帧之间的逻辑关系。

## 异质性带来的三个实战挑战

### 挑战一：维度不匹配

问你一个问题：一张 1024×1024 的图片相当于多少个文本 token？

答案是 1000-2000 个，取决于模型的分辨率设置。一段 10 秒的音频大约 160-240 个 token。这意味着你的 API 调用中，图像和音频可能占到 90% 以上的 token 消耗。

模型需要把这些不同维度的输入统一到一个序列中处理。早期融合模型的做法是把图像 patch 和文本 token 拼成一个长序列——所以图像 token 在数量上直接"淹没"文本 token。

### 挑战二：信息密度不对等

文本 token 语义密度高但缺感官细节。图像包含丰富的视觉信息但有大量冗余（背景像素）。音频在时间维度上展开，一段 10 秒的语音中可能只有 3 秒在说关键信息。

模型需要识别：
- **哪些是关键的**：图片中的人物、文字、核心物体
- **哪些是冗余的**：纯色背景、重复纹理、无声片段
- **哪些是互补的**：图片中的"红色" + 文字描述中的"紧急"——组合起来才是完整信息

### 挑战三：语义鸿沟

"猫"这个文字和一个"有胡须、尖耳朵、毛茸茸的生物"的图片——描述同一个概念，但表示完全不同。文字是抽象符号，图片是具体的视觉特征。

CLIP 训练时做的一件事就是"填平这条鸿沟"——让"猫"的文字向量和猫的图片向量在空间中靠在一起。这条鸿沟填得越好，模型在"我说一个词你找对应的图"这类任务上的表现就越好。

## 模态组合成熟度速查

不同模态组合的成熟度差别很大，理解这一点能帮你做选型判断：

| 组合 | 成熟度 | 代表产品 | 实际表现 |
|------|--------|---------|----------|
| 文 → 图 | ✅ 非常成熟 | DALL-E 3、Midjourney、Flux、SD3.5 | 日常生产可用，差距在版权和合规 |
| 文 → 视频 | ⚠️ 在迭代 | Sora、Kling、Runway Gen-3、Pika | 短视频可用，长视频和时间一致性还在进步 |
| 文 → 音乐 | ⚠️ 在迭代 | Suno、Udio | 短曲目质量惊人，长曲目逻辑还有问题 |
| 图+文 → 文 | ✅ 非常成熟 | GPT-5.5、Claude Opus 4.8 | 文档分析、视觉问答已上生产 |
| 语音 → 语音 | ✅ 已可用 | Realtime API、Gemini Live | 语音助手可用，打断处理还需完善 |
| 图 → 动作 | ⚠️ 可用 | Computer Use（Claude）、Operator（OpenAI） | 简单操作稳定，复杂流程还不行 |
| 视频流理解 | 🔄 早期 | Gemini 长视频 | 短片段可用，长时间流不稳定 |

**注意两件事**：第一，生成侧（文→图/视频/音乐）和理解侧（图+文→文）的技术路线完全不同——生成用扩散模型，理解用 Transformer。第二，成熟度差异很大。文生图已经是成熟产品，文生视频还在快速迭代中，选型时不要用一个领域的标准去套另一个领域。
