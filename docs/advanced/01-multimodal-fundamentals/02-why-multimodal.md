# 为什么需要多模态

单模态方案在 2024 年之前是唯一选择——那时候没有成熟的多模态 API，你只能把图片转成文字喂给 LLM，把语音转成文字再处理。但 2026 年的今天，继续用这套流水线方案的意义已经很小了。

先看几个真实场景，从最常见的开始。

## 场景一：文生图——你已经离不开多模态

你用 Midjourney 生成海报、用 DALL-E 3 做配图、用 Stable Diffusion 生成电商素材。这是最典型的多模态应用：输入文字，输出图像。

这个场景本身就是多模态的证明——你不需要把"需求"先转成什么中介格式，模型直接从文字跳到图像。Flux、SD3.5、Imagen 3 在 2026 年已经把图像质量推到几乎以假乱真的水平，而它们和 ChatGPT、Claude 的区别不仅是一个做理解一个做生成——底层架构完全不同（Transformer 家族 vs 扩散模型家族），优化目标完全不同（理解准确率 vs 图像 FID 分数），计费方式完全不同（按 token 还是按张数）。

## 场景二：分析一份带图表的 PDF 合同

你让 Agent 分析一份合同。合同里有文字条款、图表数据、手写签章、公司 logo。如果你的 Agent 只能处理文本，流水线长这样：

```
PDF → OCR 提取文字 → 视觉模型描述图表 → 签名验证
                                              ↓
                                     LLM 综合 → 回答
```

每个环节跑在自己的 API 上、各自计费、各自有延迟、各自会出错。

### 问题一：信息逐级丢失是结构性的

每一环的转换都在丢失信息。OCR 从图片中提取文字时，字体颜色、排版位置、手写批注的语义、盖章的大小和位置——这些要么变成文字被简化掉，要么完全丢弃。

等所有中间结果拼到一起交给 LLM 时，原始图片中的"这一页的右下角有一个红色的圆形印章"已经被简化成"有印章"三个字（如果 OCR 识别到"印章"这个词的话）。而"印章的位置是否覆盖了签名区域"这种问题，流水线方案天生答不了——它没有保留空间信息。

### 问题二：错误会叠加

OCR 把"张三"识别成"张二"——这个错误会传递到最终回答，而且难以定位。多模型拼接的调试复杂度是 O(n)——每个环节都可能出错，你需要在 n 个日志中追溯。

### 问题三：上下文断裂

模型看不到"原始文档"什么样。它只能看到拆解后的文字碎片。"合同的第二页第三行"这种问题，流水线方案需要额外开发定位追踪机制。

## 原生多模态方案的做法

多模态模型的处理方式完全不同——一次 API 调用，原始文档直接输入，模型在内部做跨模态理解。

```
PDF 合同 → 多模态模型（一次 API 调用）→ 文字内容 + 图表数据 + 签章位置 → 回答
```

核心区别不是"少调几个 API"，而是模型内部的跨模态注意力机制让文字 token 可以直接"关注"到图片中对位的 patch 向量。模型能说清"第三页右下角的签章处有盖章，红色，圆形"——因为它确实在内部"看到"了那个位置。

## 场景三：语音交互的故事

你做一个语音助手。流水线方案：

```
用户语音 → ASR（Whisper）→ 文字 → LLM 处理 → 文字 → TTS（语音合成）→ 语音回复
```

问题：
- ASR 把"明天下午三点"识别成"明天下午三店"→ 整个流程白做
- 串行延迟：ASR(1s) + LLM(1s) + TTS(1s) = 3s+，不适合实时对话
- 丢失了声调、情绪、停顿——用户用焦虑的语气说"这个方案真的可行吗"，ASR 提取的是中性文字，LLM 无法感知焦虑

多模态方案（Realtime API）：

```
用户语音 → 一次 API 流式调用 → 语音回复
```

延迟降到亚秒级，语气和情绪信息不丢失，用户打断时能从当前状态接上。

## 场景四：文生视频和音乐

文生视频（Sora、Kling）和音乐生成（Suno、Udio）这类应用天然就是多模态的——输入文字，输出视频或音乐。它们没有一个"流水线方案"的替代选项。你不能用 OCR + LLM + ASR 拼出一个文生视频功能。

这就是多模态的另一个面向：**有些能力只有原生多模态才能做**。理解侧的流水线方案只能说"质量差一点但勉强能用"，视觉生成侧的扩散模型没有替代方案。

## 三组数字帮你决策

不是所有场景都需要原生多模态。以下三组对比帮你判断：

| 场景 | 流水线方案 | 原生多模态方案 | 差距 |
|------|-----------|---------------|------|
| 分析带图 PDF | 3 次 API，总延迟 3-6s | 1 次 API，延迟 1-2s | 延迟 50-70% 降低，质量更高 |
| 语音助手 | ASR+LLM+TTS，延迟 3s+ | Realtime API，延迟 <1s | 延迟 70%+ 降低，支持情绪感知 |
| 文生图/视频/音乐 | ❌ 没有替代方案 | 原生支持 | 没有可比性，根本做不了 |

流水线方案在 2024 年之前是必要的妥协。2026 年它还在两个场景下有意义：
1. **极端预算限制**——每个环节用最便宜的模型拼起来可能比原生多模态 API 便宜
2. **需要控制单个环节**——为了合规或特殊需求，必须使用特定的 OCR 或 ASR 服务

其他场景，原生多模态方案在质量、延迟、维护成本三个维度上都更优。
