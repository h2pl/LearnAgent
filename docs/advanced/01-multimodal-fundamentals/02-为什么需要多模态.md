# 为什么需要多模态

先看一个真实场景。

你让 Agent 帮你分析一份 PDF 合同。合同里有文字条款，也有图表数据、手写签章、公司 logo。如果你的 Agent 只能处理文本，它需要走这样一条流水线：

```
PDF → OCR 提取文字 → 视觉模型描述图表 → 签名验证模型
                                            ↓
                                   LLM 综合所有文本结果
                                            ↓
                                       回答用户
```

这条流水线能跑通，但问题很明显。

## 流水线方案三个致命问题

### 信息逐级丢失

每一轮转换都在丢失信息。OCR 提取文字时，手写批注的语义、签章的位置、图表的空间结构——这些信息要么完全丢失，要么被简化成一段文字描述。等所有中间结果拼到一起交给 LLM 时，原始文档中一半以上的信息可能已经没了。

### 管线脆弱

五个不同的服务串行工作：PDF 解析、OCR、视觉分析、签名验证、LLM 综合。任何一个环节出错（OCR 识别错误、视觉模型没认出图表类型），整个回答的质量都会受影响。更糟糕的是，你很难定位错误来自哪个环节。

### 上下文断裂

模型看不到文字和图表的原始对应关系。一张合同表格的"数字"和"项目名称"在空间上是相邻的，但经过 OCR 和文本提取后，这种空间关系完全丢失了。模型只能靠拼接后的文字猜测哪个数字对应哪个项目。

## 原生多模态方案是怎么做的

多模态模型的做法完全不同：

```
PDF 合同图片 → 多模态模型（一次 API 调用）
                    ↓
              文字内容 + 图表数据 + 签章位置
                    ↓
              综合分析回答
```

**不需要拆成多个步骤。** 一张合同图片直接传给模型，它在内部同时"看到"文字内容、图表数据、签名位置，然后直接输出综合分析。

这不是"把 OCR 和 LLM 合并到一个 API"那么简单——模型在内部做跨模态注意力计算时，文字 token 可以直接"关注"到图片中对应位置的 patch 向量，信息不会因为中间转换而丢失。

## 三组对比帮你判断

| 场景 | 流水线方案 | 原生多模态方案 |
|------|-----------|---------------|
| 分析带图表的 PDF | OCR + 图表描述模型 + LLM，3 次 API | 一次 API 调用 |
| 语音助手 | ASR → LLM → TTS，3 次 API，串行延迟 | 一次 API 调用，流式返回 |
| 截图 + 语音混合提问 | 语音转文字 + 图片+文字给 LLM，2 次 API | 一次 API 调用 |

流水线方案的价值在 2024 年之前还很明显——那时没有成熟的多模态模型。但 2026 年的今天，除非你有极端特殊的需求（比如必须本地部署且多模态模型不支持你的场景），否则原生多模态方案在质量、延迟、维护成本三个维度上都占优。

## 什么时候可以不用多模态

有些场景不需要多模态。一个简单的判断方法：**如果你的输入去掉非文本信息后仍然完整，就只需要文本模型。** 如果去掉后丢失了关键信息，那多模态就是必要的。
