你好，我是江小湖。前几篇逐个讲了多模态的各项能力：看懂图片、听懂语音、生成图像。但在真实的复杂场景中，这些能力很少单独使用——你更可能面对的是"用户发了一张报错截图并用语音描述了问题"这种多模态混合输入。

跨模态推理就是让模型在这种多模态混合场景中做出正确判断。这是多模态 AI 的最高级形态，也是 2026 年最前沿的挑战之一。

## 什么是跨模态推理

先看几个例子：

**场景 1：合同审核**
你上传了一份合同扫描件（图片），同时粘贴了对方的邮件说明（文字），问 Agent："邮件里说的付款条件和合同里写的一致吗？"

Agent 需要同时理解图片中的文字内容（视觉）和邮件文本（语言），然后做比对推理。

**场景 2：技术支持**
用户发来一张应用报错截图，同时语音说"我刚才点了保存按钮就变成这样了"。

Agent 需要理解截图中的错误信息（视觉），结合用户的操作描述（语音），定位问题原因。

**场景 3：数据分析**
你上传一张销售数据的柱状图，问"Q3 和 Q1 的差距主要在哪几个产品线？"

Agent 需要从图表中读取具体数值（视觉），做数值比较（推理），然后用文字总结（语言）。

这些场景的共同特点：**单一模态无法完成任务，模型必须在多个模态之间建立联系并做综合推断。** 这就是跨模态推理。

## 2026 年的能力边界

跨模态推理在 2026 年已经有了显著进步，但远不是所有场景都可靠。

### 已经可靠的场景

- **图片 + 文字 → 描述/比较**："这张图和那段描述一致吗？"
- **截图 + 问题 → 定位**："这个 UI 报错的原因是什么？"
- **图表 + 问题 → 数据读取**："图中 2024 年的收入是多少？"
- **多张图 → 对比**："这两张设计稿的差异在哪？"

这些任务的共同特点是：模态之间的对应关系明确，推理链条短（1-2 步）。

### 仍然不太可靠的场景

- **精确数值推理**：从图表中读取多个数值后做复杂计算（加减乘除还行，统计分析就悬了）
- **长链推理**：需要跨多个模态做多步推导（看图→查数据→推理→再生成图）
- **矛盾检测**：当不同模态的信息存在细微矛盾时（合同文字和盖章不一致）
- **专业领域**：医学影像 + 病历的综合诊断、法律文件 + 证据的交叉分析

### 能力边界的关键因素

跨模态推理的可靠性受几个因素影响：

1. **模态对齐质量**：对齐越好，跨模态推理越准（参见[对齐与表示学习](../01-multimodal-fundamentals/08-alignment-and-representation.md)）
2. **推理链长度**：步骤越多，错误累积越多
3. **信息密度差异**：图片和文字承载的信息量差异很大时，模型容易忽略低密度模态的信息
4. **模型融合方式**：早期融合 > 交叉注意力 > 晚期融合（跨模态推理能力递减）

## 多步推理链：多模态场景下的规划

和单模态推理不同，多模态的多步推理需要模型在每一步决定"用哪个模态的信息"。

### 一个实际的推理链示例

用户任务："分析这个项目的 README、首页截图和最近的 GitHub 活跃度，给我一个是否值得使用的建议。"

Agent 的推理链：
```
Step 1: 读 README（文本）→ 提取核心功能和定位
Step 2: 看首页截图（图像）→ 评估 UI 质量和用户体验
Step 3: 查 GitHub API（工具调用）→ 获取 star 数、最近提交频率、issue 响应时间
Step 4: 综合三个模态的信息 → 做推荐判断
```

每一步都涉及不同模态，最终回答需要把三种模态的信息综合起来。

### Agent 的规划策略

对于多模态的复杂任务，Agent 应该：

1. **分解任务**：把多模态任务拆成单模态子任务
2. **逐模态处理**：每个子任务用最适合的模态/模型处理
3. **汇聚结论**：把所有子任务的结论放在一起做最终判断
4. **交叉验证**：如果不同模态给出了矛盾结论，标记出来让用户确认

这种"分治→汇聚"的模式比试图一次性处理所有模态更可靠——尤其在推理链较长的时候。

## 多模态记忆：跨会话的关联

多模态推理不只发生在单次对话中。在长期使用中，Agent 需要跨会话关联多模态信息。

这和[第 9 章记忆管理](../../09-memory-management/README.md)讨论的问题一脉相承，但多了一个维度：记忆的模态。

### 场景举例

- 上周用户给 Agent 看了一张架构图，讨论了系统瓶颈
- 这周用户发了一段新代码，问"这个改动对上次说的瓶颈有帮助吗"

Agent 需要关联"上周的架构图"和"这周的代码"——跨模态、跨会话的关联推理。

### 实现思路

**多模态记忆存储**：
- 把每次交互中的图片、语音、文本都存入记忆系统
- 图片可以存原始文件，也可以存视觉模型生成的文字描述（更省空间）
- 建立跨模态的索引：每张图片关联当时的对话上下文

**检索策略**：
- 文本查询 + CLIP 语义检索双通道
- 用户提到"上次那张图"时，通过时间线索 + 语义相似度定位
- 对于频繁引用的图片，保留原始分辨率；其他的降级为描述

**记忆压缩**：
- 图片记忆太占空间，需要对历史图片做摘要（存文字描述而非原图）
- 音频记忆可以存转录文本 + 关键片段

这和文本记忆管理的原则一致（参见[第 9 章](../../09-memory-management/README.md)），只是存储和检索都需要扩展到多模态。

## 评估跨模态推理质量

怎么知道你的多模态推理系统"好不好用"？

### 构建回归测试集

和文本场景类似，你需要一个针对多模态推理的回归测试集：

```python
test_cases = [
    {
        "image": "contract_scan_001.png",
        "text": "邮件说付款条件是 net 30，和合同一致吗？",
        "expected_contains": ["net 30", "一致"],
        "expected_not_contains": ["矛盾", "不一致"],
    },
    {
        "image": "error_screenshot_003.png",
        "text": "这个报错怎么解决？",
        "expected_contains": ["permission denied"],
        "evaluation": "llm_judge",
    },
]
```

### 评估指标

**自动指标**：
- **跨模态一致性（Cross-modal Consistency）**：模型的回答是否同时考虑了所有模态的信息
- **事实准确率**：从图片/图表中读取的数值是否正确
- **完整性**：是否遗漏了某个模态中的关键信息

**LLM Judge 评估**：

```python
# 用 LLM Judge 评估跨模态推理质量
evaluation_prompt = """
请评估以下回答是否正确融合了图片和文字两种信息：

[图片描述]: {image_description}
[用户问题]: {user_question}
[模型回答]: {model_answer}

评分标准：
- 5分：完全正确，充分利用了图片和文字信息
- 3分：基本正确，但遗漏了某个模态的部分信息
- 1分：有明显错误，或完全忽略了某个模态
"""
```

### 持续监控

在生产环境中，多模态推理的"静默失败"比单模态更常见——模型可能只用了文字信息就给出回答，看起来对但实际上忽略了图片中的关键信息。

监控策略：
- 对每次多模态请求记录"模态使用率"（模型是否真的用了所有模态）
- 定期用 LLM Judge 对线上请求做抽样评估
- 设立阈值：跨模态一致性低于某个值时告警

## 总结

跨模态推理是多模态 AI 的最高级形态：

- 它要求模型同时处理多种模态并做综合推断，单一模态无法完成
- 2026 年在短链推理上已经可靠，长链推理和专业领域仍有不足
- Agent 的多步推理策略是"分解→逐模态处理→汇聚结论"
- 多模态记忆需要扩展到存储和检索多种模态的信息
- 评估跨模态推理需要专门的回归测试集和 LLM Judge
