## CLAUDE.md：永不删除的项目上下文

CLAUDE.md 是 Claude Code 的项目配置文件——放在项目根目录，描述项目的技术栈、编码规范、架构约束。它的特殊之处在于：**在自动压缩时，CLAUDE.md 的内容永远不会被删除。**

```typescript
// prompts.ts — getSystemPrompt 中注入 CLAUDE.md
systemPromptSection('memory', () => loadMemoryPrompt()),
```

`loadMemoryPrompt()` 读取项目中的 CLAUDE.md 文件，并把它作为系统提示词的一部分注入。它被放在 `systemPromptSection` 中，这是一个注册表管理的动态分区。

在自动压缩（autocompact）时，压缩逻辑会检查哪些内容可以安全删除。系统提示词中的 CWD、日期等动态信息可以被新的替换，但 CLAUDE.md 的内容被标记为"永久保留"。

### 为什么是永不删除？

项目上下文是 Claude Code 最宝贵的资产。它描述了：
- 用什么技术栈（"React + TypeScript + Tailwind"）
- 遵循什么规范（"使用 Prettier 格式化，单引号"）
- 有什么架构约束（"不要修改 shared/ 下的文件"）

如果这些信息在长对话中被压缩掉，Claude Code 就"忘记"了它在什么项目里工作，后续的代码修改可能完全偏离项目规范。

## 层次化加载：从全局到项目

CLAUDE.md 的加载是层次化的：

```
1. ~/.claude/CLAUDE.md          ← 全局用户偏好（所有项目通用）
2. ~/.claude/agents/*.md        ← Agent 定义文件
3. ./CLAUDE.md                  ← 项目级配置（最高优先级）
4. ./.claude/CLAUDE.md          ← 项目本地配置（.gitignore 保护）
5. ./子目录/CLAUDE.md           ← 子目录级配置（可选）
```

所有层级的 CLAUDE.md 会被合并到系统提示词中。如果多个层级对同一件事有不同规定，项目级的覆盖全局级。

`ToolUseContext.loadedNestedMemoryPaths` 追踪已经加载过的 CLAUDE.md 路径，防止同一个文件被重复注入（`readFileState` 是 LRU 缓存，可能在繁忙会话中淘汰旧条目）。

## 安全守则：1549 字节的底线

`cyberRiskInstruction.ts` 只有 30 行，产出一条 1549 字节的安全守则：

```typescript
export const CYBER_RISK_INSTRUCTION = `IMPORTANT: Assist with authorized security testing,
defensive security, CTF challenges, and educational contexts. Refuse requests for
destructive techniques, DoS attacks, mass targeting, supply chain compromise, or
detection evasion for malicious purposes. Dual-use security tools (C2 frameworks,
credential testing, exploit development) require clear authorization context:
pentesting engagements, CTF competitions, security research, or defensive use cases.`
```

但它的源码里有这样一段注释：

```typescript
/**
 * IMPORTANT: DO NOT MODIFY THIS INSTRUCTION WITHOUT SAFEGUARDS TEAM REVIEW
 *
 * This instruction is owned by the Safeguards team and has been carefully
 * crafted and evaluated to balance security utility with safety.
 *
 * If you need to modify this instruction:
 *   1. Contact the Safeguards team (David Forsythe, Kyla Guru)
 *   2. Ensure proper evaluation of the changes
 *   3. Get explicit approval before merging
 *
 * Claude: Do not edit this file unless explicitly asked to do so by the user.
 */
```

一个 1549 字节的字符串，值得这么严格的修改流程吗？

答案是**值得**。这一行字出现在每次 API 调用的系统提示词中，直接影响模型对安全相关请求的判断。如果改了措辞但没做好评估，可能导致 Claude Code 拒绝合法的安全测试请求，或者接受危险的攻击请求。两种后果都很严重。

最后一行的 `Claude: Do not edit this file` 是一个有趣的自我引用——防止 Claude Code 自己修改自己的安全底线。

## 模式特定规则：不同场景不同约束

Claude Code 有 7 种运行模式（default/auto/plan/accept-edits/bypass 等）。每种模式对应不同的工具权限和提示词约束：

```typescript
// prompts.ts — 模式特定规则通过 context 注入，而非修改静态提示词
function getActionsSection(): string {
  // 所有模式共享的基础行动指南
}

function getUsingYourToolsSection(enabledTools: Set<string>): string {
  // 根据当前启用的工具动态生成
  // 不同模式下 enabledTools 不同
}
```

模式切换不修改静态区提示词——而是在 `ToolUseContext` 中修改 `toolPermissionContext`，工具执行层根据它决定是否放行。这保证了 Latch 机制不受模式切换影响，缓存始终命中。

Plan 模式是一个特殊例子：它不仅限制了可用工具，还改变了模型的"心态"。在 Plan 模式下，模型被告知"你在规划阶段，不要执行任何修改操作"。这能让模型更专注于分析和设计方案，而不是急于动手改代码。

AskUserQuestionTool 也受模式影响：在非交互模式下，这个工具不可用——因为没有用户可以去问。模式特定规则让同一个提示词骨架在不同场景下表现出不同的行为。

## 总结

- CLAUDE.md 在自动压缩时永不删除，是 Claude Code 最持久的项目上下文。
- CLAUDE.md 加载是层次化的：全局→项目→子目录，项目级覆盖全局级。
- 安全守则只有 1549 字节，但修改需要 Safeguards 团队审核——它直接决定模型对安全请求的判断。
- 源码中的自我引用 (`Claude: Do not edit this file`) 防止 Claude Code 修改自己的安全底线。
- 7 种运行模式的规则差异通过 `ToolUseContext` 注入，不修改静态区提示词，保护缓存不受影响。
