你好，我是江小湖。前面讲过 LLM 的本质是"预测下一个词"。但就是这个简单的机制，涌现出了令人惊叹的多样化能力。这篇把每项能力摊开看——不是泛泛地说"它能做 X"，而是从架构机制出发，说清楚"能力从哪来、能做到什么程度、哪里做不到"。

## 文本理解与生成

这是 LLM 最基础也最强的能力，也是所有其他能力的底层基础。

### 具体能做到什么

LLM 能做摘要、翻译、润色、扩写、改写风格、提取要点、格式转换。质量已经接近甚至超过普通人类水平，在某些特定任务上（如多语言翻译、风格改写）甚至超过专业人类编辑。

```
输入: "请用最通俗的语言解释什么是 Transformer"

输出: "想象你在一个图书馆查资料。你拿着一本书（当前词），
       但你需要知道其他书里有没有提到相关的东西。
       Transformer 就是让你同时翻开所有书，快速对比，
       找到跟你当前这本书最相关的段落。
       它不是逐页阅读，而是一眼扫过去，靠注意力找到关键信息。"
```

再看一个风格改写的例子：

```
原文: "Transformer 架构于 2017 年由 Google 提出，采用自注意力机制取代了
      传统的循环神经网络，实现了并行计算。"

改成给小学生看: "有一种很聪明的读书方法，是 2017 年一个叫 Google 的公司发明的。
                 以前读书只能一个字一个字读，很慢。
                 这种新方法可以一眼看整页书，找到最重要的部分，快多了。"

改成学术论文风格: "Vaswani et al. (2017) 提出了 Transformer 架构，
                  该架构以自注意力机制（self-attention）为核心，
                  摒弃了传统的循环神经网络（RNN）范式，
                  实现了序列处理的并行化，显著提升了计算效率。"
```

### 能力的架构基础

"预测下一个词"这个训练目标，在万亿级 Token 的预训练语料上执行后，会产生一个重要的结果：**为了准确预测下一个词，模型的参数必须编码语言的统计规律**。

具体来说，预训练语料（后文会详细展开）包含网页、书籍、论文、代码、对话、新闻等几乎所有人类写过的文字形式。模型的参数在梯度下降优化过程中，为了最小化预测损失，不得不捕捉以下统计规律：

- **词序概率分布**：哪些词倾向于出现在哪些词之后——这就是语法结构的统计表征
- **文体概率分布**：学术论文、口语对话、代码注释等不同文体各自有不同的词序列分布模式。当 Prompt 要求"学术风格"时，模型的预测概率分布会偏向学术文本的统计模式
- **语义向量关系**：Embedding 层（后文会详细展开）将语义关系编码为向量空间的几何关系。"国王"和"女王"的向量差近似于"男人"和"女人"的向量差——这不是模型"理解"了性别概念，而是统计规律在向量空间中的自然涌现

**关键理解**：模型不是在"运用语言规则"，而是万亿级数据 + 数十亿参数 + 预测目标的优化过程，将语言的统计规律**编码进了参数矩阵**。风格改写不是在套模板，而是模型在条件概率分布 `P(下一个词 | 上文 + "请用学术风格")` 中采样——这个分布已经被训练数据塑造成了学术文本的概率形态。

### 能力的边界在哪里

**擅长**：
- 同语言内的改写、摘要、润色
- 主流语言对之间的翻译（中英、英法、英日等）
- 保持语义的同时改变风格或格式

**不擅长**：
- **小众语言翻译**：训练数据少的语言（如藏语、威尔士语），参数中编码的统计规律不充分，翻译质量明显下降
- **需要精确数字的摘要**：如果原文有具体数据（"2024 年营收 3.7 亿"），摘要可能把数字改错或省略——数字在 BPE 分词后是 Token 序列而非数值，模型处理的是 Token 概率而非数字本身
- **创意写作的"灵魂"**：模型能写出结构完整、语言流畅的文章，但缺乏真正独特的观点和情感深度。它更像是一个优秀的"文字工匠"，而不是"思想家"

### 对开发者的影响

凡是涉及"把文字变成另一种文字"的任务——翻译、改写风格、提取要点、生成报告、格式化输出——LLM 几乎都是可靠的第一选择。但在需要精确数字、小众语言、或深度创意的场景，需要人类审核或补充。

## 逻辑推理

LLM 能解决逻辑谜题、进行常识推理、分析因果关系。但这项能力有一个重要的分水岭：**普通模型和推理模型的表现差距巨大**，而这个差距的根源是计算预算。

### 普通模型的推理能力

普通 LLM（如 GPT-4o、Claude Sonnet）能处理简单的逻辑推理，但在多步骤推理上容易出错：

```
问: "小明比小红大 3 岁，小红比小华小 2 岁，小华 10 岁。小明几岁？"

普通模型可能直接答: "小明 15 岁"
← 错误。它跳过了中间步骤，直接猜测了一个看起来合理的数字。

正确答案: 
  小红 = 10 - 2 = 8 岁
  小明 = 8 + 3 = 11 岁
```

**架构层面的原因**：标准的自回归模型（前面讲过）每生成一个 Token，计算量是固定的——一次 Transformer 前向传播。简单问题（"中国的首都是哪里？"）一次前向传播就够了，因为答案在训练数据中出现频率极高，概率分布非常集中。但多步推理需要**更多的计算步骤**——先算小红，再算小明——一次前向传播的计算量不足以完成两步运算，模型只能在概率分布中"猜"一个看起来合理的答案。

类比：这就像让一个人做数学题但**只准写一行答案**，不准写草稿。简单口算没问题，多步运算就容易出错——不是他不懂数学，而是不给中间步骤的计算空间。

### 推理模型：思维链（Chain of Thought）

2022 年之后，研究者发现了一个简单但有效的方法：让模型**先生成推理步骤，再给出最终答案**。这被称为**思维链（Chain of Thought, CoT）**。

```
问: "小明比小红大 3 岁，小红比小华小 2 岁，小华 10 岁。小明几岁？"

推理模型会先写:
  "小红比小华小 2 岁 → 小红 = 10 - 2 = 8 岁
   小明比小红大 3 岁 → 小明 = 8 + 3 = 11 岁"
  然后给出答案: "小明 11 岁"
← 正确
```

**思维链为什么有效**——从架构机制看：每生成一个中间步骤的文本（如"小红 = 8 岁"），模型就完成了一次完整的前向传播。这个中间结果作为新的上下文进入下一步预测时，模型的自注意力机制可以直接"看到"这个中间结果，相当于给它提供了一步计算的依据。思维链的本质是**用生成的文本串联多次前向传播**，把一次不够的计算拆成多次。

类比：允许写草稿了。每写一行草稿就是一次完整的前向传播计算，写三行草稿就是三次计算，准确率自然提升。

2024 年之后，OpenAI 的 o1、DeepSeek-R1 等**推理模型**把 CoT 内化到了训练过程中。这些模型在生成最终答案之前会自动生成内部推理过程（有时是用户不可见的"思考 Token"），本质上是被训练成**主动花更多 Token 来做推理**——用更多的自回归生成步骤换取更高的计算深度。

### 推理模型 vs 普通模型的取舍

| 维度 | 普通模型（GPT-4o） | 推理模型（o1、R1） |
|------|-------------------|-------------------|
| 推理能力 | 中等，多步推理易出错 | 强，复杂逻辑任务显著优于普通模型 |
| 速度 | 快 | 慢（多了推理步骤） |
| Token 消耗 | 少 | 多（推理过程也要计费） |
| 适用场景 | 简单问答、文本生成、日常对话 | 数学证明、代码调试、复杂分析 |

**核心区别**：推理模型不是"更聪明"，而是**花了更多 Token（= 更多计算步骤）来推理**。就像考试时，一个学生每题只写答案（普通模型），另一个学生先写草稿再写答案（推理模型）——后者的准确率更高，但用时更长。

**对开发者的影响**：**不要无脑用推理模型**。简单任务用普通模型（快、便宜），复杂推理任务才用推理模型（准、但贵）。选错了要么浪费钱，要么得不到正确答案。

## 代码生成与分析

代码生成是目前 LLM 创造商业价值最大的领域之一。代码之所以特别适合 LLM 处理，不是因为模型"懂编程"，而是因为**代码的特性恰好匹配自回归预测模型的优势**。

### 具体能做到什么

```
输入: "写一个 Python 装饰器，记录函数的执行时间"

输出:
  import time
  from functools import wraps

  def timer(func):
      @wraps(func)
      def wrapper(*args, **kwargs):
          start = time.perf_counter()
          result = func(*args, **kwargs)
          elapsed = time.perf_counter() - start
          print(f"{func.__name__} 执行耗时: {elapsed:.4f}s")
          return result
      return wrapper

  # 使用示例
  @timer
  def slow_function():
      time.sleep(1)
      return "done"
```

除了写代码，LLM 还能：

- **阅读和解释代码**：给它一段不熟悉的代码，它能解释每一步在做什么
- **找 bug**：描述你遇到的错误，它能分析代码找出可能的问题
- **写单元测试**：给它一个函数，它能生成覆盖各种边界情况的测试
- **代码重构**：让它把一段代码改成更清晰、更高效的版本
- **跨语言转换**：把 Python 代码翻译成 Java，或把 SQL 翻译成 Pandas 代码

### 代码为什么适合 LLM

代码相比自然语言有几个关键特性，这些特性让自回归预测模型在代码领域表现更稳定：

- **严格的语法约束**：代码是形式语言，语法有明确规则（编译器会拒绝不合法的代码）。训练数据中的代码几乎全是语法正确的，模型编码的概率分布自然集中在合法代码序列上
- **高度重复的模式**：装饰器、设计模式、API 调用方式、错误处理套路——这些模式在 GitHub 上反复出现数百万次，模型对它们的预测概率非常集中
- **结构化程度高**：代码有明确的层级结构（函数、类、模块），Token 之间的依赖关系比自然语言更固定。自注意力机制（[Transformer 内部原理](./07-transformer-internals.md)会详细讲）能精准捕捉这些结构化依赖

**关键理解**：代码的概率分布比自然语言更"尖锐"。同一个意思在自然语言中可以有无数种表达方式（概率分布宽），但实现同一个功能的正确代码只有有限的几种模式（概率分布集中）。模型在代码上的预测准确率因此更高——这不是它"懂编程"，而是代码的结构化特性让预测任务更容易。

### 能力的边界在哪里

**擅长**：
- 主流语言的常见任务（Python、JavaScript、Java、TypeScript）
- 标准库和流行框架的 API 使用
- 算法实现、数据结构、设计模式

**不擅长**：
- **小众库和过时 API**：训练数据中出现频率低的库，模型编码的统计规律不充分，可能生成不存在的函数或过时的 API 调用（这是代码领域的幻觉——后文会详细展开）
- **复杂架构设计**：跨多个文件、涉及系统整体设计的任务，需要全局性的规划能力。自回归模型逐 Token 生成，缺乏对整体架构的"鸟瞰"能力
- **私有代码和内部系统**：公司的内部 API、私有库、业务逻辑完全不在训练数据中，模型的参数里没有这些信息的概率分布。这类任务需要结合 RAG 或提供充分的上下文

### 对开发者的影响

代码生成是 LLM 最成熟的能力之一。对于日常开发中的"写个函数、改个 bug、加个测试"这类任务，LLM 已经是强大的生产力工具。但对于涉及系统架构、私有代码、或需要长期维护的大型项目，LLM 更适合做"助手"而不是"主导者"——人类负责架构决策，LLM 负责具体实现。

## 上下文学习（少样本学习）

这是 LLM 最实用的能力之一，也是 Agent 开发中最常用的技术：**不需要重新训练模型，只在 Prompt 里给几个例子，它就能适配一种新的输出格式或处理逻辑。**

### 零样本 vs 少样本

```
【零样本（不给例子）】
Prompt: "把下面这句话转成 JSON 格式：小明今年 25 岁，住在北京"
输出: {"名字": "小明", "年龄": 25, "住址": "北京"}
← 格式可能不稳定，字段名可能每次都不一样

【少样本（给 2 个例子）】
Prompt: "
示例1: 小红今年 20 岁，住在上海
→ {"name": "小红", "age": 20, "city": "上海"}

示例2: 小李今年 30 岁，住在深圳
→ {"name": "小李", "age": 30, "city": "深圳"}

请处理: 小明今年 25 岁，住在北京
→"
输出: {"name": "小明", "age": 25, "city": "北京"}
← 格式完全一致，字段名稳定
```

### 为什么少样本有效

这个能力的来源是 **Transformer 的自注意力机制**（后文会详细展开）。

自注意力让每个 Token 都能"看到" Prompt 中所有其他 Token 的信息。当 Prompt 中包含示例时，模型处理到新输入时，注意力层可以同时关注示例的输入和输出，计算出它们之间的映射关系，然后把这个关系应用到新的输入上。

从概率的角度看：零样本时，模型的输出分布很宽（各种格式都有可能）；给了 2-3 个示例后，自注意力机制让模型的条件概率分布 `P(输出 | 示例1 + 示例2 + 新输入)` 变窄——输出被"锚定"到了示例展示的格式上。

这还有一个重要的架构对比：RNN（前面讲过）处理长文本时，前面的信息会在逐层传递中衰减，难以利用远处的示例。Transformer 的自注意力直接建立任意两个 Token 之间的连接，无论距离多远，信息都不会衰减——这就是为什么 Transformer 天然支持上下文学习，而 RNN 很难做到。

### 少样本 vs 微调

很多新手遇到"模型输出格式不稳定"时，第一反应是"我需要微调模型"。但实际上，**2-5 个示例通常就能让输出格式稳定下来**，比微调快得多、便宜得多。

| 维度 | 少样本学习 | 微调（SFT） |
|------|-----------|------------|
| 成本 | 几乎为零（只是多几个 Token） | 高（需要训练数据 + GPU 时间） |
| 速度 | 即时生效 | 需要数小时到数天 |
| 适用场景 | 控制输出格式、学习处理模式 | 注入领域知识、稳定控制复杂任务 |
| 局限 | 示例太多会占满上下文窗口 | 需要大量高质量标注数据 |

**对开发者的影响**：**先试 Few-shot，再考虑微调**。绝大多数"格式不稳定"的问题，2-5 个示例就能解决。只有当 Prompt 已经很长（示例占满了上下文）、或者任务太复杂（示例无法覆盖所有情况）时，才考虑微调。

## 指令遵循与角色扮演

通过 System Prompt，你可以让 LLM 扮演一个特定的角色，并严格遵循你的格式和行为要求。这是 Agent 开发的核心配置机制。

### 具体能做到什么

```
System Prompt: "你是一个严厉的代码审查员。
               对于每段代码，只输出 3 个问题，每个问题不超过 15 个字。
               不要解释，不要建议，只列出问题。"

User: "def add(a, b): return a + b"

LLM: "1. 缺少类型注解
     2. 缺少返回值说明
     3. 未处理异常输入"
```

System Prompt 可以控制：

- **身份和角色**：代码审查员、心理医生、SQL 专家、翻译官
- **输出格式**：只输出 JSON、只用列表、每个回答不超过 50 字
- **行为边界**：不回答政治问题、不生成暴力内容、不确定时说"我不知道"
- **语气和风格**：严厉、温和、幽默、学术

### 指令遵循的架构基础

这个能力来自两个训练阶段（后文会详细展开），它们各自在模型参数中建立了不同的机制：

**SFT 阶段**（指令微调）：使用大量人工编写的 `(指令, 回答)` 数据对进行训练。这一阶段通过梯度下降，在模型参数中建立一种条件概率映射——当输入文本的统计特征匹配"指令"模式时，输出的概率分布偏向"按指令格式回答"而非"续写上文"。

后文会讲到，Base 模型（只经过预训练、未经 SFT）不会对指令做出回应——它会续写你的句子而不是回答问题。SFT 改变了模型的条件概率分布，让"指令→回答"这个模式的概率被大幅提升。

**RLHF/DPO 阶段**（人类对齐）：用人类偏好数据进一步优化模型。偏好评分作为训练信号——符合指令的回答得高分，不符合或有害的回答得低分。模型的参数在这个反馈信号下继续调整，"遵循指令"的生成模式被强化，"忽视指令"的模式被抑制。

**角色扮演的机制**：预训练数据包含大量人物对话、小说角色、历史人物的文本。当 System Prompt 设定"你是莎士比亚"时，这个条件文本通过自注意力机制注入到每一层的计算中——与古典英语风格相关的 Token 在预测概率分布中获得更高权重。模型不是"变成"了莎士比亚，而是在条件概率分布 `P(下一个词 | "你是莎士比亚" + 用户输入)` 中采样，这个分布偏向古典英语风格的文本。

### 能力的边界在哪里

**擅长**：
- 稳定的格式控制（JSON、列表、特定长度）
- 明确的角色扮演（专家、助手、特定风格）
- 简单的行为约束（不回答某类问题）

**不擅长**：
- **复杂的条件逻辑**：如果 System Prompt 包含太多"如果 A 则 X，如果 B 则 Y，如果 C 则 Z"，模型容易混淆或遗漏条件——注意力在多个条件之间分散，无法像程序一样严格执行分支逻辑
- **绝对的安全边界**：即使 System Prompt 明确说"绝不回答 X"，经过精心构造的 Prompt 注入攻击仍可能绕过这个限制。System Prompt 本质上只是上下文中的一段文本，模型在某些情况下可能给这段指令文本分配较低的注意力权重
- **长期一致性**：在非常长的对话中，随着对话历史的 Token 越来越多，System Prompt 在上下文中的占比越来越小，注意力权重可能衰减，模型的行为可能逐渐偏移回默认分布

### 对开发者的影响

System Prompt 是 Agent 的核心配置工具。一个好的 System Prompt 应该：

- **明确身份**：第一句话就定义角色
- **具体约束**：用数字和格式要求（"不超过 50 字"而不是"简短回答"）
- **示例示范**：给 1-2 个示例展示期望的输出格式
- **边界声明**：明确什么不做（"不确定时说不知道"）

但不要把 System Prompt 当成万能的安全机制。对于关键的安全约束（如不泄露敏感信息），需要在应用层做额外的验证和过滤。

## 总结

这篇从架构机制出发，展开了 LLM 的五项核心能力：

- **文本理解与生成**：海量语料 + 预测下一个词 → 参数编码了语言的统计规律，擅长改写、翻译、摘要，但不擅长精确数字和深度创意
- **逻辑推理**：普通模型多步推理易出错（一次前向传播的计算量有限），推理模型用思维链串联多步计算显著提升但更慢更贵，按需选用
- **代码生成与分析**：代码的结构化特性让预测概率分布更集中，主流语言和常见任务非常可靠，小众库和复杂架构需要人类引导
- **上下文学习**：自注意力机制让模型能同时看到所有示例并捕捉映射模式，2-5 个示例就能稳定输出格式，比微调快得多便宜得多，优先使用
- **指令遵循与角色扮演**：SFT + RLHF 在参数中建立"指令→回答"的条件概率映射，System Prompt 是有效的配置工具，但本质上只是上下文中的条件文本，不是万能的安全机制

**核心洞察**：LLM 的每项能力都不是它"理解"或"学会"了什么，而是架构机制（自注意力、自回归生成）和训练过程（海量数据上的参数优化）的必然涌现。理解这些机制，你才能设计正确的 Agent 方案——在该信任它的地方信任它，在该给它加辅助的地方加辅助。
